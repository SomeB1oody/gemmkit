# Dot Kernels and the Deep-K Twin

Most kernel families consume the contraction one depth step at a time: load a column of the packed A panel, broadcast one element of packed B, and issue one FMA (or widening multiply-add) per accumulator register. Two AVX-512 extensions break that rhythm by folding several depth steps into a single instruction: VNNI's `vpdpbusd` multiplies 4 consecutive `i8` depth steps into each of 16 `i32` lanes, and AVX-512 BF16's `vdpbf16ps` folds 2 consecutive `bf16` depth steps into each `f32` lane. An instruction that consumes several depth steps at once wants those steps adjacent in memory, and (for floats) it changes how the accumulation rounds, so gemmkit gives dot kernels their own kernel families and ISA tokens rather than hiding them behind a branch in the shared microkernel. This page walks through the two seams that carry them, the two concrete dot kernels, and the deep-contraction route that both narrow families share.

## Why dot instructions get their own families

A dot kernel differs from its widen sibling in exactly two places, and each lands on a different extension axis of the engine (see [Scalars and Kernel Families](Scalars_and_Kernel_Families.md) for the family/token split). The pack layout is a family concern: `KernelFamily::pack_lhs`/`pack_rhs` take no ISA parameter, so a different interleave must key off the family, which is why `Bf16DotGemm` is a sibling of `MixedGemm<bf16>` rather than a branch inside it. The inner loop is a token concern: only a CPU that actually has `vpdpbusd` or `vdpbf16ps` can run it, so the instruction lives behind a `KernelSimd` method that only dot-capable tokens override. The concrete dot families are `IntGemmVnni` (and its requantizing variant `IntGemmVnniQ`) with 4 depth steps per instruction, and `Bf16DotGemm` with 2, plus the two f32-output twins covered at the end of this page.

## DEPTH_MULTIPLE and the k-group pack

A family that folds `Q` depth steps per instruction declares `const DEPTH_MULTIPLE: usize = Q` (default 1). The contract, spelled out in `gemmkit/src/kernel.rs`, is that the family's packers write panels of `width * kc.next_multiple_of(Q)` elements, depth-padding the tail, and the driver strides packed panels by the same padded depth so both sides stay in lockstep. For every ordinary family `DEPTH_MULTIPLE = 1` and all the `next_multiple_of` calls degenerate to identities.

The layout itself comes from one shared routine, `pack_kgroup_panels` in `gemmkit/src/pack.rs`, the single source of truth for the interleave index math. Plain `pack_panels` stores a panel depth-major: at each depth step, `width` contiguous leading elements (`mr` rows for the LHS, `nr` columns for the RHS). `pack_kgroup_panels` instead groups the depth axis `Q` at a time so that one lane's `Q` consecutive depth values become contiguous: within a panel, group `g`, lane `i`, in-group position `t` lands at offset `g*width*Q + i*Q + t`. That is exactly the shape one dot instruction reads: a 64-byte A register covers `LANES` rows times `Q` contiguous depth elements, and a B group broadcasts `Q` contiguous depth values of one column as a single 32-bit load.

Two more details of the shared packer matter. It takes a per-element transform `xform`, identity for bf16 and the `+128` bias for VNNI's A operand, and it fills every pad position (leading positions past the block and depth positions past `kc`) with `xform(0)`, so the pad is always consistent with the live elements. And because the interleaved layout cannot be read in place, every dot family sets `FORCE_PACK_LHS`/`FORCE_PACK_RHS`, overriding the driver's cost-based pack decision: a dot kernel always pays the pack, which is precisely the cost the gates described below hedge against. A byte-level oracle in `pack.rs`'s tests reimplements the layout naively and checks the real routine reproduces it bit-for-bit across width tails, depth tails, and strided sources.

On the consuming side, `KernelSimd::dot_accumulate` is the seam the families call instead of the widen-FMA loop. Its default body is `unreachable!`: only a dot-capable token overrides it, and only a dot family ever calls it. `Avx512Vnni` and `Avx512Bf16` are distinct tokens from `Avx512F` because `#[target_feature]` is per-token: `_mm512_dpbusd_epi32` needs an `avx512vnni` codegen context that `Avx512F::vectorize` (only `avx512f`) does not establish. The method receives the real, unpadded `kc` and reads `ceil(kc / Q)` instruction groups from the depth-padded panels; any signedness or bias correction is applied internally, so the accumulators hold the true `sum_k(A*B)` on return. The reason the fold lives on this dedicated seam rather than on the generic `accumulate_tile` is documented on the trait: fusing depth steps reshapes the accumulation rounding, which `accumulate_tile`'s contract forbids.

## i8 through vpdpbusd

`vpdpbusd` computes an unsigned-times-signed dot: u8 from the first operand, i8 from the second. GEMM wants signed-times-signed, and the fix is algebraic rather than per-element: the LHS pack offsets every byte by `+128` into the unsigned domain (`vnni_a_xform` in `gemmkit/src/kernel/int.rs`, using the constant `VNNI_A_BIAS = 128` defined once in `gemmkit/src/simd.rs` so pack and correction can never drift apart). Since `sum_k((A+128)*B) = sum_k(A*B) + 128*sum_k(B)`, the kernel recovers the true product by subtracting a per-column correction `128 * sum_k(B[k][j])`. `Avx512Vnni::dot_accumulate` computes those column sums with a small scalar pass over the signed packed B panel before the vector loop, then subtracts the splatted correction from every accumulator at the end. The pads cooperate: the A pad is `xform(0) = 128`, whose contribution the correction cancels exactly, and the B pad is `0`, which contributes nothing to the product or the column sums.

Because `i32` accumulation wraps and wrapping addition is associative modulo `2^32`, regrouping the sum into quads plus the bias correction equals the ascending-`k` widen sum bit-for-bit: `IntGemmVnni` and the widen `IntGemm` produce identical output on every input. The ISA choice can therefore never change an `i8` result, which is a stronger property than the [reproducibility contract](Design_Goals_and_the_Big_Picture.md) requires.

That freedom to swap kernels mid-flight is what the small-parallel fallback gate exploits. The VNNI pack is mandatory on both operands, and on a small multi-threaded problem that pack barrier dominates the compute it saves. So when the selection was automatic, the parallelism is `Rayon(n)` with `n != 1`, and `m*n*k` falls below `GEMMKIT_I8_VNNI_MIN_PAR_MNK` (default `768^3`), `dispatch/int.rs` hands the call to the in-place widen kernel instead; serial runs and large parallel runs keep VNNI, and a forced `GEMMKIT_REQUIRE_ISA=avx512vnni` disables the gate entirely (force must run exactly that kernel). The prepacked-RHS path also bypasses the gate, for two reasons: a k-quad-interleaved buffer is only consumable by the VNNI family, and the pack barrier the gate hedges against was already amortized once at prepack time; since VNNI's RHS pack is otherwise mandatory on every call, prepacking is a bigger win there than for any in-place-capable kernel.

## bf16 through vdpbf16ps

`Bf16DotGemm` is the floating sibling: `DEPTH_MULTIPLE = 2`, both operands packed k-pair-interleaved (each pair is one 32-bit `__m512bh` element), and `dot_accumulate` issuing one `vdpbf16ps` per accumulator per pair-step. Everything downstream of the accumulation, the alpha fold and the widen-read/narrow-store epilogue, is shared verbatim with `MixedGemm<bf16>` through the common `mixed_epilogue` helper, and the family keeps `OUT_IS_ACC = false` so the whole contraction accumulates in `f32` and rounds to bf16 once.

The numeric story differs from VNNI in one essential way: `vdpbf16ps`'s fused 2-term dot rounds differently from two separate widen-FMAs, so the dot kernel's result is only tolerance-equal to the widen path, not bitwise. That is exactly what the engine's consistency bar allows: results must be reproducible under a fixed input, environment, and configuration, not bitwise-identical across kernel choices. The dot kernel is fully deterministic, and serial, parallel, and prepacked runs all share the same kernel and pack layout, so they reproduce each other bit-for-bit. Consequently there is no size gate on this path: auto-selection prefers `Bf16DotGemm` whenever the CPU reports `avx512bf16` (the selection ladder's comment notes it roughly doubles bf16 throughput), and unlike VNNI there is no small-parallel fallback. The three special-path reroutes are the only exceptions: gemv, `small_mn`, and small-`k` shapes deliberately stay on `MixedGemm<bf16>`'s widen seam, because a tiny or degenerate output folds nothing and the dot pack's depth padding is pure loss there (the `i8` dispatch reroutes its tiny shapes to the widen kernel for the same reason).

## The deep-K problem

`OUT_IS_ACC = false` buys single-rounding at a structural price: the driver runs `kc = k`, one depth panel spanning the entire contraction, instead of the cache-model `kc` slices every homogeneous family gets (see [Blocking and the Cache Model](Blocking_and_the_Cache_Model.md)). The RHS micropanel a microtile call reads is then `nr * k * sizeof(N)` bytes, and once that outgrows L2, every one of the `m/mr` microtile calls per column strip streams it, alongside the even larger `mr * k` LHS micropanel, from L3 or DRAM. The cliff is measurable and sharp: on the Zen5 9950X (AVX-512F, `nr = 12`), `deep_k_engage_bytes`'s comment in `gemmkit/src/cache.rs` records `f16`/`bf16` throughput still near peak at `k = 16384` (a 384 KiB micropanel) and collapsed at `k = 32768` (768 KiB, about 0.75x the 1 MiB L2).

The engage gate in `gemmkit/src/dispatch/mixed.rs` compares that micropanel size against a byte threshold:

```rust
let engage_deep_k = NR
    .checked_mul(t.k)
    .and_then(|x| x.checked_mul(core::mem::size_of::<N>()))
    .is_some_and(|bytes| bytes > crate::cache::deep_k_engage_bytes());
if engage_deep_k {
    run_deep_k_twin::<N, Fam::Twin, S, MR_REG, NR>(simd, &t, par, ws);
    return;
}
```

The threshold is the `GEMMKIT_DEEP_KC_BYTES` knob taken verbatim when non-zero; the default `0` derives it as half the effective per-worker L2 capacity, a deliberate choice over a full-L2 gate, which on the measured machine would engage only past `k ~ 43000` and miss the cliff entirely (the L2/2 gate catches `k = 32768` at 2.8x and `k = 65536` at 3.6x faster for `f16`, while leaving `k = 16384` and below on the single panel, where the twin is within noise). The `checked_mul` chain fails closed: a broadcast operand can pass validation with a logically absurd `k`, and an overflowing size must fall through to the single panel, whose own pack sizing then rejects the problem, rather than engage a twin that would multi-slice that `k` forever.

## The f32-output twin

Above the gate, dispatch does not run the narrow family at all. A small `DeepKTwin` trait maps each narrow family to its f32-output twin, `MixedGemm<N> -> MixedGemmF32<N>` and `Bf16DotGemm -> Bf16DotGemmF32`, families whose only type change is `Out = f32 = Acc`. That one change flips `OUT_IS_ACC` back to its `true` default, so the driver's ordinary multi-slice K blocking applies unchanged and every slice's panels are L2-resident again, which is the entire point. The pack layout and the accumulation loop are the narrow family's verbatim: `MixedGemmF32` reuses `pack_panels` and the shared widen-FMA helper, `Bf16DotGemmF32` reuses `pack_kgroup_panels` and `dot_accumulate`, and since the accumulation helpers only touch the input side of the `KernelSimd` seam, the accumulator they produce is byte-for-byte what the narrow family would compute.

The twin runs with `alpha = 1`, `beta = 0` into an `m x n` column-major `f32` scratch drawn from a dedicated `Workspace` (deep-K is by definition a large-`k` regime, so one `m*n` f32 allocation is negligible, and keeping it separate leaves the pooled packing workspace free for the twin driver). Afterward, one vectorized sweep computes `narrow(alpha*scratch + beta*widen(C))`, replicating `mixed_epilogue`'s arithmetic operation for operation, including the same `store_out` narrowing.

What makes the route more than an approximation is how slices chain. The twin's microkernel seeds its accumulator registers from the scratch through a third `KernelSimd<N, N, f32, f32>` seam (`twin_seed` in `gemmkit/src/kernel/mixed.rs`): on an accumulate slice it loads the running partial into the registers and continues the ascending-`k` chain, rather than summing the slice from zero and adding afterward. A store and reload of an `f32` is exact, so the multi-slice sum is the single-panel sum merely split at slice boundaries, and for `beta` in `{0, 1}` the deep-K result is byte-for-byte the single-panel result. For a general `beta` it holds to tolerance instead, for a mundane reason: the single panel fuses `beta*C + AB` in one FMA on full tiles but combines it unfused on edge tiles, so no single sweep can match both. Serial and parallel remain bit-identical in every case, since the twin driver's blocking is thread-count independent and the sweep is elementwise. For the dot twin there is one extra alignment rule: the driver rounds the blocking `kc` up to `DEPTH_MULTIPLE`, so an interior slice boundary never splits a k-pair; a split pair would zero-pad mid-contraction and regroup the fused dot, and with the rule only the final short tail is padded, exactly as in the single-panel case.

Three routes deliberately keep the single panel: shallow `k` below the gate, the fused-epilogue entries, and the prepacked-RHS path, where a `DEPTH_MULTIPLE > 1` buffer requires the whole contraction to be one depth slice (a hard assert in the driver, because violating it would misaddress micropanels silently). The parity claims are tested directly: `gemmkit/tests/deep_k_narrow.rs` toggles `GEMMKIT_DEEP_KC_BYTES` (`1` forces the twin at any `k`, `usize::MAX` forces the single panel) and checks byte-for-byte equality for `beta` in `{0, 1}`, tolerance for general `beta`, on whichever ISA the host selects; the knob itself is documented with the rest in [Tuning Knobs](../gemmkit-guide/Tuning_Knobs.md).
