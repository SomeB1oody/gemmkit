# Special Paths

The register-tiling driver described in [Life of a GEMM Call](Life_of_a_GEMM_Call.md) is built around one assumption: there is enough work per output tile to amortize packing, blocking, and a full `MR x NR` register accumulator. Some shapes break that assumption badly. A matrix-vector product has no tile reuse at all; a `k = 4` product finishes before the pack pays for itself; a `8 x 8 x 100000` contraction would spend most of the driver's effort multiplying zero padding. Layer L6 (`gemmkit/src/special/`) reroutes exactly these shapes to dedicated kernels. Three properties hold for every reroute: it hides behind the same public entries (`gemm` and friends never expose which path ran), it is threshold-gated through [tuning knobs](../gemmkit-guide/Tuning_Knobs.md) so a mis-calibrated gate can be moved or disabled without a recompile, and it keeps the library's reproducibility contract — for the shapes below, bit-identical output across worker counts.

The gates live at the top of each per-type dispatch entry, in a fixed order. From `run_typed` in `gemmkit/src/dispatch/float.rs` (trimmed):

```rust
// gemmkit/src/dispatch/float.rs (run_typed, trimmed)
if (t.n == 1 || t.m == 1) && core::cmp::min(t.m, t.n) <= tuning::gemv_threshold() {
    gemv::run_typed_epi::<T, S, Identity>(/* user frame, before orientation */);
    return;
}
orient_transpose(&mut t);
if small_mn_eligible(&t) || small_mn_pack_eligible(&t) {
    small_mn::run_epi::<T, S, Identity>(/* horizontal dot kernel */);
    return;
}
if t.k <= tuning::small_k_threshold() {
    small_k::run::<FloatGemm<T>, S, MR_REG, NR>(/* one depth panel, in place */);
    return;
}
driver::run::<FloatGemm<T>, S, MR_REG, NR>(/* the general blocked driver */);
```

gemv fires before orientation normalization, in the user's coordinate frame; the other gates run on the oriented problem. Each special path also exists in fused-epilogue form, so a `gemm_fused` call takes the same route its plain twin would — that contract is the subject of [Epilogue Fusion](Epilogue_Fusion.md).

## gemv: the memory-bound edge

A shape with `m == 1` or `n == 1` (`gemmkit/src/special/gemv.rs`) does `2k` flops per output element against `k` matrix elements read once: it is memory-bound, and the whole game is minimizing DRAM traffic, not scheduling FMAs. Both orientations reduce to one core routine by viewing the matrix (transposed for `m == 1`) as `rows x k` times a `k`-vector. The gate is shape, not size: `min(m, n)` is 1 for any gemv shape, so the `GEMMKIT_GEMV_THRESHOLD` knob it is compared against acts as an off switch (set it to 0 to force such shapes onto the general driver, which is also correct) rather than a size cap.

Parallelism follows a bandwidth model, not the compute ramp: `Parallelism::resolve_bandwidth` stays serial below an LLC-derived byte floor (one core already gets the full cache bandwidth there), then steps straight to a bandwidth cap derived from the core count, because a few workers is the worst point on a bandwidth scaling curve. Output rows are partitioned across workers in panels whose grain is a multiple of the SIMD width, each row's full `k`-reduction stays inside one worker, and there are no split reductions or cross-worker merges — so the result is bit-identical for any worker count, a stronger property than the engine-wide reproducibility bar.

Within a worker's range, the layout picks one of four strategies. A column-major matrix takes the axpy shape, which has two deliberately bit-identical variants differing only in memory traffic: a **register-blocked output** form that holds a panel of output rows in SIMD registers across the whole `k`-sweep (matrix and output each read exactly once), and a **plain column-outer** form that re-reads the output each column but streams the matrix as one perfectly contiguous read. The choice is output cache residency, computed in `output_register_block`: register-block when the output (`rows * sizeof`) outgrows a fraction of the last-level cache (so the plain form's re-reads would hit DRAM) *and* `k` is at or below `GEMMKIT_K_STREAM_MAX` (default 32; past that, the register-blocked form's `k` concurrent column streams thrash the hardware prefetcher). Both variants run the same ascending-`k` fused accumulation per element and the same per-row SIMD/scalar split, so switching the gate never changes a single bit — it only changes speed. A row-major matrix takes a dot form (rows register-blocked in groups of 4 to overlap FMA latency chains, each row still the shared fixed-order `dot_contiguous` reduction), and fully strided operands fall to a scalar loop.

The mixed-precision twin `run_mixed` (feature `half`) serves `f16`/`bf16` gemv with the same row partition and the same bit-identity argument, but every load widens to `f32` through the `KernelSimd<N, N, f32, N>` seam, the reduction runs in `f32`, and the result rounds to the narrow type exactly once at the store. One asymmetry follows from that single-rounding discipline: the mixed axpy always register-blocks the output, because the plain column-outer form would re-read and re-write the narrow output per column group — rounding once per group instead of once per element. And the mixed *fused* gemv deliberately does not route here at all: the float fused gemv fuses by re-reading each stored output and mapping it in place, which is bit-exact only because the float output *is* the accumulator; a narrow output has already rounded at the store, so re-read-and-map would round twice. Rather than thread the epilogue through every widen store, the mixed fused entry keeps gemv shapes on the general driver, which already applies the epilogue in `f32` before the single narrowing (`gemmkit/src/dispatch/mixed.rs`).

## small-k: one depth panel, nothing to amortize

At tiny `k` (`gemmkit/src/special/small_k.rs`), the whole product is a single depth panel, so the driver's cache-blocking model, workspace carving, and above all its A/B packing are pure setup: every packed element would be read once. The route computes `C <- alpha*A*B + beta*C` directly over the family's microkernel with `kc = k`, reading A and B **in place** — no packing, no blocking, no workspace traffic — while inheriting the family's widen/rounding semantics for free, since it is generic over `KernelFamily`. The gate is `k <= GEMMKIT_SMALL_K_THRESHOLD`, an arch-split machine calibration (default 16 on x86, 8 on aarch64, where the narrower NEON tile packs cheaply enough that the driver wins sooner).

The in-place read has preconditions, and the route defers to the driver (still correct, just differently scheduled) when they fail: the microkernel wants unit-stride LHS rows, so A must be column-major (`rsa == 1`); a `FORCE_PACK_*` family (complex, whose pack *transforms* the data into planar form) cannot be read in place by construction; and `k` past a hard `SMALL_K_MAX = 32` would overflow the one stack buffer the route does use — a zero-padded panel for the bottom partial row-tile, which must be packed because the microkernel always loads full `mr` rows. Work is partitioned over output tiles, each tile one full `k`-pass by one worker, with the bandwidth-model worker cap (the `m*n` output write dominates at small `k`): bit-identical across worker counts, same as gemv.

## small-mn: horizontal dots for tiny outputs

When both `m` and `n` sit far below the microtile but `k` is long (`gemmkit/src/special/small_mn.rs`), the driver would pad the tiny row and column tiles up to full `MR x NR` microtiles and compute mostly padding. This route instead computes each output element as one horizontal SIMD dot `C[i,j] = alpha*<A[i,:], B[:,j]> + beta*C[i,j]`, streaming along the contraction. The output is register-blocked into `4 x 4` tiles of accumulators, so 16 independent FMA chains stay in flight across the `k`-sweep while each A-row and B-column loads once per tile — the same latency-hiding trick as gemv's dot form, generalized to a small grid. The dims gate is `m, n <= GEMMKIT_SMALL_MN_DIM` (default 16; 32 on aarch64, where the zero-copy tier dominates through 32) with `k` above the small-k threshold, so the two small-shape routes partition the `k` axis between them.

The kernel needs both operands unit-stride along `k`: A rows contiguous (`csa == 1`, row-major A) and B columns contiguous (`rsb == 1`, column-major B). That is the zero-copy tier. The two most common layouts each miss exactly one side — all-row-major misses B, all-col-major misses A — so a second, mutually exclusive gate (`k > GEMMKIT_SMALL_MN_PACK_MIN_K`, default 16) engages a pre-pack tier: `prepack_operands` copies **only the failing operand** into a `k`-contiguous workspace scratch and runs the identical kernel over it with unit strides. The copy touches `m*k` (or `n*k`) elements against the `m*n*k` dot work — a tax of roughly `1/n` (or `1/m`) that the horizontal win dwarfs, so a strided small-`m,n` shape still beats falling back to the driver's padded microtiles. The scratch rounds each line up to an *odd* number of cache lines (`packed_line_stride`): the natural stride `k` would map every packed line to the same L1 set whenever `k` is a power of two, collapsing the re-reads (measured about 3x below the driver at `k >= 1024` before the fix). Because the pre-pack is a pure reorder — the same values in the same per-line order — the packed route is bit-identical to the eligible-layout route. Mixed (`f16`/`bf16`, widen to `f32`, round once per cell) and integer (`i8 -> i32`, wrapping and therefore bit-exact against the driver) siblings share the tiling, the pre-pack helper, and the reproducibility argument.

## batched: orchestration, not a kernel

Batched GEMM (`gemmkit/src/special/batched.rs`) is deliberately not a new kernel: every element re-dispatches through the full single-GEMM engine, so a batch composes with the driver, gemv, small-k, and small-mn automatically. What the layer adds is a schedule, chosen once per call by `Parallelism::resolve_batch`:

- **`BatchParallel`** — enough elements to fill the workers: each worker runs whole GEMMs serially and cache-hot, and the batch pays one fork/join instead of one per element. This is the model for the motivating workload of many small matrices.
- **`SequentialInternal`** — few but large, DRAM-bound elements: loop the batch, giving each element the full engine parallelism in turn. On x86 the split engages when an element spills the per-core L2; on aarch64 (shared cluster L2, high unified bandwidth) it engages when the per-batch-worker share `elem_bytes / batch` exceeds `GEMMKIT_SEQ_INTERNAL_BYTES_PER_WORKER` (default 128 KiB). This plan splits an element across workers, so it is gated to `m, n > 1` shapes — the driver / small-k / small-mn routes reduce every output within one worker and agree serial-vs-parallel bit-for-bit under the thread-independent blocking, while gemv is held only to reproducibility and is therefore excluded.
- **Serial** — below the total-work gate, or without usable threads.

Since elements are independent and the serial and batch-parallel plans never split one, a batch is bit-identical across worker counts on those plans regardless of the per-element route. The strided-batched entries (`gemm_batched`, and `gemm_batched_fused`, which threads one shared epilogue through the same skeleton) share one schedule implementation so the plain and fused forms cannot drift. For batches whose elements differ in shape, the pointer-array form `gemm_batched_ptr_unchecked` takes a slice of `GemmProblem` descriptors — each with its own dimensions, strides, and pointers — under the simpler `resolve_batch_flat` policy (whole GEMMs to workers, never the per-element split, since there is no uniform residency to test); `gemm_batched_slice` is its safe, validated twin. The user-facing view of all of this is in [Batched GEMM](../gemmkit-guide/Batched_GEMM.md) and [Small Shapes and GEMV](../gemmkit-guide/Small_Shapes_and_GEMV.md); the worker-count machinery these paths lean on is described in [Parallel Execution](Parallel_Execution.md).
