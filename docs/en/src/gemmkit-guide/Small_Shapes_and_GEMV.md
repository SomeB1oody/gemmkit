# Small Shapes and GEMV

The register-blocking driver at the heart of gemmkit is built for products with enough work per output tile to amortize packing, cache blocking, and a full `MR x NR` accumulator. Some shapes break that premise outright: a matrix-vector product has no tile reuse at all, a `k = 4` contraction finishes before the pack pays for itself, and an `8 x 8 x 100000` product would spend most of the driver's effort multiplying padding. For these the driver is the wrong tool, and the engine quietly routes around it.

There is nothing to opt into. The same `gemm` (and `gemm_i8`, `gemm_fused`, `gemm_map`) entry inspects the shape and strides at the top of dispatch and reroutes to a special-case kernel when one fits, falling through to the general driver otherwise. Every reroute stays behind the same public entry, keeps the library's reproducibility contract (bit-identical output across worker counts), and is gated by a [tuning knob](Tuning_Knobs.md) you can move or disable without recompiling. You never call these paths directly; you just benefit from writing the shape naturally instead of hand-rolling a dot loop.

## gemv: the memory-bound edge

A shape with `m == 1` or `n == 1` is a matrix-vector product. Each output element does only `2k` floating-point operations while reading `k` matrix elements, so the arithmetic is trivial and the entire game is minimizing DRAM traffic. The dedicated gemv path (`m == 1` and `n == 1` reduce to one core routine, viewing the matrix, transposed for `m == 1`, as `rows x k` times a `k`-vector) is correct for every layout and vectorized for the contiguous ones.

Two properties matter to a caller. First, gemv is **bit-identical across worker counts**: each output element is computed in a single pass over `k` by one worker, and the output-row partitioning never splits an element's reduction, so adding threads never changes a bit. Second, gemv has **its own parallel policy**. Because it is bandwidth-bound, the worker count is capped by a bandwidth model rather than the compute ramp the general driver uses: past the few cores that saturate DRAM, more workers stop helping and would only add fork/join and shared-cache contention. Two knobs expose this, `gemv_parallel_bytes` (a byte floor below which gemv stays single-threaded because the data is cache-resident) and `gemv_thread_cap` (a hard ceiling on gemv workers), alongside `gemv_threshold`, which is effectively an on/off switch since a gemv shape always has `min(m, n) == 1`.

## The small-k path

When the contraction `k` is small (at or below `small_k_threshold`, default 16 on x86 and 8 on aarch64), the whole product is a single depth panel, and the driver's packing has nothing to amortize because every packed element is read exactly once. The small-`k` route (skinny and low-depth shapes: gevv, rank-`k` updates, tall-skinny products) computes `C` directly over the family's microkernel, reading `A` and `B` **in place**, unpacked, in one pass. It inherits the family's widen / bias / conjugate / rounding behavior for free and is bit-identical to the serial run for any worker count. It needs a column-major `A` (unit-stride rows, `rsa == 1`); when that does not hold, packing rarely amortizes at tiny `k`, so the route defers to the general driver, which is still correct.

## The small-m,n path

The mirror case is a tiny output with a long contraction: both `m` and `n` far below the microtile (at or below `small_mn_dim`, default 16) but `k` long. The driver would pad the tiny row and column tiles up to a full microtile and compute mostly padding. This route instead computes each output as a horizontal dot, `C[i,j] = alpha * <A[i,:], B[:,j]> + beta * C[i,j]`, streaming SIMD along the contraction with no blocking or orientation machinery, and register-blocks the small output grid so several independent FMA chains stay in flight.

The horizontal kernel needs both operands unit-stride along `k`: `A` rows contiguous (`csa == 1`, a **row-major A**) and `B` columns contiguous (`rsb == 1`, a **column-major B**). When both hold, the route reads `A` and `B` in place with zero copies, the fast path. The two most common layouts each miss exactly one side (all-row-major fails `rsb`, all-col-major fails `csa`), so an internal pre-pack copies **only the failing operand** once into `k`-contiguous scratch and runs the same horizontal dot over it. That copy is `m*k` (or `n*k`) reads against the `m*n*k` product, a roughly `1/n` (or `1/m`) tax the horizontal win dwarfs, so a strided small-`m,n` shape still beats falling to the driver's padded microtile. The pre-pack tier engages once `k` clears its own knob, `small_mn_pack_min_k` (default 16), separate from the zero-copy tier's `small_k_threshold`. This path is likewise bit-identical across worker counts, computing each output as one fixed-order reduction on a disjoint tile.

## Practical guidance

Prefer the ordinary entries over a hand-written dot loop. If your problem is a matrix-vector product, a rank-`k` update, or a grid of small outputs over a long contraction, `gemm` already carries a kernel tuned for it, complete with the bandwidth-aware threading and bit-exact reproducibility a hand loop would have to reinvent. Layout is the one lever in your hands: for the horizontal small-`m,n` path, a **row-major A and column-major B** stream unit-stride and hit the zero-copy fast path; for the small-`k` path a **column-major A** stays on the in-place route. Any other layout still works, but pays either the small-`m,n` pre-pack copy or a fall-back to the general driver.

Every threshold named here is a tunable knob, resolved per call from an argument, a programmatic setter, a `GEMMKIT_*` environment variable, or the calibrated compile-time default. If a reroute is miscalibrated for your machine or you want to force a shape onto the general driver, the [Tuning Knobs](Tuning_Knobs.md) chapter covers `gemv_threshold`, `small_k_threshold`, `small_mn_dim`, `small_mn_pack_min_k`, `gemv_parallel_bytes`, `gemv_thread_cap`, and `k_stream_max` in full. The special-path *internals* (why each kernel is shaped the way it is) are in the architecture chapter's [Special Paths](../architecture/Special_Paths.md).
