# 分块与缓存模型

驱动器最底层的微内核把一个 `MR x NR` 的 `C` 块保持在寄存器里，沿深度方向流式读入一条 `A` 微面板和一条 `B` 微面板。这个循环只有在两条输入流都来自就近的缓存时才能跑到机器峰值——而一次 GEMM 触碰的数据远超任何缓存的容量：`A` 的每个元素要在 `n` 个输出列上复用，`B` 的每个元素要在 `m` 个输出行上复用。分块（blocking）就是对这种复用的编排：把问题切开，使每个操作数块只被装入某一级缓存一次，然后在被逐出之前反复读取。`KC` 切分深度，让一次微块乘法读到的两条微面板在整个计算期间驻留 L1；`MC` 决定打包后的 `A` 宏面板大小，使它在驱动器扫过当前列块的每个列瓦片时始终留在 L2；`NC` 决定打包后的 `B` 宏面板大小，使它在每个行块轮流扫过时始终留在 L3。让面板跨越循环嵌套保持驻留正是全部意义所在——否则同样的字节要从 DRAM 里反复流过 `m`、`n` 或 `k` 遍。

许多库按微架构硬编码 `(MC, KC, NC)`；gemmkit 则在每次调用时用 BLIS 模型在 `CacheTopology::blocking`（`gemmkit/src/cache.rs`，L3 层）里从运行时探测到的缓存几何解析地推导。输入是微块尺寸 `(mr, nr)`、单个打包输入元素的字节大小和问题形状 `(m, n, k)`；输出是驱动器循环所用的 `Blocking { mc, kc, nc }` 三元组。[一次GEMM调用的生命周期](一次GEMM调用的生命周期.md)展示了每个值落在嵌套的哪一层；本页讲的是这些值如何推导、模型为何长成这个样子。

## 三个约束，三个块尺寸

### KC：两条微面板同驻 L1 且不自我逐出

每次微块调用走 `kc` 个深度步，每步读 `mr` 个打包 `A` 元素和 `nr` 个打包 `B` 元素，所以一条 `mr x kc` 和一条 `nr x kc` 的微面板必须在整个微块计算期间同驻 L1d。微妙之处在于*不自我逐出*：缓存不是一个字节池，而是按组（set）划分的若干路（way）；一条面板若把过多自己的缓存行映射到同一组上，早在总大小逼近缓存容量之前就会把自己挤出去。因此模型以行和组、而非字节来计算：先算出每条微面板一个深度步占用多少 L1 行，取合并占用不超过 L1 相联度的最大 `kc`，再抬高到 `GEMMKIT_KC_MIN` 下限（默认 512），保证小 L1 不会饿死微内核的深度扫描，最后钳制到 `k`。收尾还有一次再平衡：把 `k` 切成 `ceil(k / kc)` 个大小均匀的面板，避免最后一个深度片是碎屑。

### MC：A 宏面板占 L2，扣除 B 的份额

在一个行块内，打包后的 `mc x kc` `A` 面板要在当前列块的每个列瓦片上复用，所以它应当填满 L2——但不能全占，因为同一深度片的 `nr x kc` `B` 微面板在每次微块调用时都要流经 L2。模型先数出这条微面板占多少 L2 路，再额外保留一路备用，其余容量全部交给 `A`：`mc` 等于该容量除以 `kc`，向下取整到 `mr` 的倍数，再平衡使各行块均匀，最后被 BLIS 式硬上限 `GEMMKIT_MC_REG_PANELS * MR`（默认 8 个微块行）钳住。这个上限是标定点而非不变量——它在目前测过的每种拓扑上都实际生效，所以 `MC` 实践中就是 `MR` 的一个小倍数，L2 容量项只是余量。

### NC：B 宏面板占 L3，无 L3 时用面板数上限

有 L3 时，为途经的 `A` 流量保留一路，其余预算给打包后的 `kc x nc` `B` 宏面板：`nc` 等于该容量除以 `kc`，向下取整到 `nr` 的倍数并对 `n` 再平衡。有些机器根本不报告 L3——Apple Silicon 的簇共享 L2 就是层级的顶端——此时模型改为全 `N` 直到一个面板数上限：`nc` 取 `GEMMKIT_NC_NO_L3_PANELS * nr`（默认 512 个面板，`nr = 4` 时即 2048 列）并以 `n` 封顶。没有 L3 可以让 `B` 驻留，`B` 反正要从 DRAM 流过；这个上限只是约束共享的打包 `B` 缓冲区大小，而不是在建模驻留。

## 按打包元素、而非累加器元素计大小

`sizeof` 参数是单个**打包输入**元素的大小——驱动器传的是 `size_of::<Fam::Lhs>()`，不是累加器的大小——因为模型预算的面板是以打包 `Lhs`/`Rhs` 为单位存储的。对 `f32`/`f64` 两者相同，没有区别；但对窄类型这个区分值得实打实的深度：`i8` 每元素打包 1 字节而累加器 `i32` 是 4 字节，`f16`/`bf16` 打包 2 字节而累加器 `f32` 是 4 字节，若按累加器计大小，它们的 `kc` 和 `nc` 会被砍到缓存实际容纳量的四分之一或一半。窄类型因此获得成比例更深的块，这正是它们能在同一硬件上跑赢 `f32` 的原因之一。预打包入口用一个哨兵行数复用同一模型，使[预打包操作数](打包与工作区.md)的几何与最终的 `m` 无关。

## 把输出 tile 预取到 LLC 之外

三个块尺寸让 `A` 和 `B` 面板保持驻留，却对 `C` 只字未提：每次微 tile 调用都要对它的 `mr x nr` 输出 tile 做读-改-写，而一旦一次调用的工作集——`A`、`B`、`C` 三者的字节合计——超出单核可达的 LLC，这个输出 tile 就不再驻留缓存，它的存储要伸进 DRAM。驱动器用一次软件预取来应对，每次调用只决策一次：把工作集与 `cache::prefetch_ws_bytes`（即 `GEMMKIT_PREFETCH_MIN_BYTES` 门槛，`0` = 自动 = 单核可达的 LLC，有 L3 取 L3，否则取 L2）比较，门槛一过，就在每个输出微 tile 对应的微内核调用之前，对它发出一次 T0 预取，趁微内核还在计算时把存储将要触及的缓存行拉进 L1。这个提示沿 tile 的单位步长维度整行整行地走 64 字节缓存行（两维都带步长的 tile 没有连续行，跳过），且只在 `x86_64` 上发出（`prefetcht0`，基线 SSE，无需 feature 门控）；在任何其他目标上它降解为空，因此 aarch64 和 wasm 不受影响。它只搬动缓存行，绝不动算术，所以门槛开、关或强制时结果都逐位一致。在 Zen5 9950X 上实测（`f32`、AVX-512、每 CCD 32 MiB 的 LLC）：`2048^3`（工作集 48 MiB）并行 +1.4%、串行约 +1%，`3072^3` 并行 +3.0%，深 K 的 `2048x2048x24576` 形状并行 +2.6%；门槛以下 tile 驻留缓存，路径不变，实测只是噪声。

## 小矩阵捷径

当 `m` 和 `n` 都不超过 `GEMMKIT_TINY_BLOCK_DIM`（默认 64）时，完整模型被跳过：`kc` 取 `k` 并以 `GEMMKIT_KC` 上限（默认 512；aarch64 为 16384——实测切片越深越快）钳制，`mc` 取该深度下面板还能留在 L2 的行数（并以 `m` 本身封顶），`nc` 取 `n` 向上取整到 `nr`。整个工作集都装得进 L2 的问题不需要三级驻留分析；省下的运算花在小问题真正的瓶颈——每次调用的固定开销——上更划算。

## 探测：一条不会失败的回退链

模型的质量取决于喂给它的几何数据，而缓存几何没有可移植的查询方式。gemmkit 跑一条尽力而为的回退链，其中 `#[cfg]` 只挑选*嗅探方法*，从不决定*数值*：`#[cfg(target_arch)]` 分不清 Intel 和 AMD，虚拟机或容器又可能屏蔽 CPUID、隐藏 `/sys`，所以每个后端都返回 `Option`，链条最终落到一个不可能失败的常量上。

```rust
// gemmkit/src/cache.rs
#[cfg(feature = "std")]
fn detect() -> CacheTopology {
    // try the CPUID backend
    #[cfg(all(any(target_arch = "x86", target_arch = "x86_64"), not(miri)))]
    if let Some(t) = cpuid::detect().filter(plausible) {
        return t;
    }
    // try the sysfs backend
    #[cfg(all(target_os = "linux", not(miri)))]
    if let Some(t) = sysfs::detect().filter(plausible) {
        return t;
    }
    // try the sysctl backend
    #[cfg(all(target_os = "macos", not(miri)))]
    if let Some(t) = sysctl::detect().filter(plausible) {
        return t;
    }
    ZEN5_FALLBACK
}
```

各后端依次是：**CPUID**（`cache/cpuid.rs`）是一条指令，与操作系统无关，在容器和多数虚拟机里都可用——两家厂商都走按缓存实例枚举的拓扑叶（Intel `04h`，AMD `0x8000_001D`），它描述的是*执行核实际可达*的那块缓存：在多 die 的部件（如双 CCD 的 Ryzen）上，L3 报的是一个复合体内核真正命中得到的那块（9950X 上为 32 MiB），而不是整片总量——这正是该值所有消费者需要的语义。没有该叶的老 AMD 部件或屏蔽它的虚拟化环境回退到传统的 L1（`0x8000_0005`）和 L2/L3（`0x8000_0006`）叶，其中 L3 大小以 512 KiB 为单位报告且为整片总量，16 路相联度甚至无法编码。**Linux sysfs**（`cache/sysfs.rs`）用纯 `std::fs` 解析 `/sys/devices/system/cpu/cpu0/cache/index*/`——在 x86 Linux 上是回退项（针对屏蔽 CPUID 的虚拟化环境），在没有 CPUID 指令的 aarch64 Linux 上则是主要来源。**macOS sysctl**（`cache/sysctl.rs`）通过一个两行的 `extern "C"` 块调用 `sysctlbyname`（不引入 `libc` 依赖），优先读 Apple Silicon 的按性能级键（`hw.perflevel0.*`，即 P 核），Intel Mac 的扁平键作为回退；`sysctl` 不暴露相联度，于是采用保守的典型值——这是安全的，因为模型对相联度只需近似值并用 `.max(2)` 钳底。链条的终点是 `ZEN5_FALLBACK`，一个在 Ryzen 9950X 开发机上标定的静态默认值：48 KiB / 12 路 L1d、1 MiB / 16 路私有 L2、32 MiB / 16 路 L3。

有两道守卫让这条链不止是“按顺序尝试”。`plausible` 过滤不合理的读数——任何一级小于 4 KiB、缓存行小于 16 字节或相联度为零都会让整个后端出局，因此被屏蔽的叶不可能用零值毒害分块。而且探测每个进程至多运行一次：`Machine::current()` 用 `OnceLock` 把拓扑连同操作系统页大小（`getpagesize`，校验为 4 KiB 到 2 MiB 之间的 2 的幂）一起记忆化；页大小驱动 [打包与工作区](打包与工作区.md)里描述的 LHS 打包步长闸门。`no_std` 构建完全跳过探测，使用 Zen5 回退值和 4 KiB 页。

## `shared_by`：对驱动器实际放置的数据的争用

每个 `Level` 携带 `bytes`、`assoc`、`line` 和一个推导字段 `shared_by`，后者把该级容量除成模型实际预算的 `effective_bytes`。把硬件的核共享数直接填进去看似自然——但那是错的。`shared_by` 建模的是*对驱动器实际放在该级的数据的按 worker 争用*，而驱动器的放置方案是：每个 worker 的 `A`/`B` 微面板放 L1d，每个 worker 私有的 `A` 宏面板放 L2，唯一一份共享的 `B` 宏面板放 L3。

这个放置方案直接决定了取值。L1d 按核私有，整个容量都服务于一个 worker 的微面板：`shared_by = 1`。L3 在硬件上被所有核共享，但驱动器放在那里的是一条所有 worker *共同读取*的面板——同一份字节，不是每 worker 一份拷贝——所以整级容量都属于这一条面板，`shared_by` 同样是 `1`；若按原始核数去除，预算会缩水 16 到 32 倍，把 `NC` 无谓地打垮。只有 L2 真正装着按 worker 私有的数据，所以只有 L2 使用*物理核*的 L2 共享度：私有 L2 的机器（主流 x86、Neoverse）取 `1`，核簇共享一个 L2 的机器（Apple Silicon）取簇大小——那里多个 worker 的私有 `A` 面板确实在争抢同一批路。每个后端必须*推导*这个值而不是照抄原始计数：sysfs 用 L1d 共享列表读出的 SMT 度去除 L2 原始 `shared_cpu_list` 计数（超线程兄弟不能重复计），sysctl 读 `hw.perflevel0.cpusperl2`，CPUID 后端直接置 `1`（x86 的 L2 按物理核私有）。在 x86 和 Graviton 上整套机制归约为全 1；它是为簇共享 L2 的机器准备的——在那里，它决定了模型是按 worker 实际分到的 L2 分块，还是按一个要五方共用的 L2 分块。

## 线程数能挪动什么，不能挪动什么

`blocking` 没有线程数参数，对 `KC` 和 `NC` 而言这个缺席是承重的：两者都只依赖机器和问题本身，所以串行运行和 32 worker 运行推导出同样的 `KC` 与 `NC`，进而是同样的深度片、以及每个输出元素同样的固定顺序深度链。`MC` 则是驱动器*确实*会为并行度调整的那一个分块维度。当宽 worker 数会让扁平任务列表太浅——每个 worker 分不到几个块，运行的尾部退化成一群空等最后几块的 worker——驱动器便缩小 `MC`，切出更多行块把列表加深（即并行作业深度下限，详见[并行执行](并行执行.md)）。所以面板边界和扁平任务列表已经*不再*严格与 worker 数无关。

尽管如此，逐位一致依然成立，因为这次缩小是无关数值的。`MC` 始终保持为 `MR` 的倍数，所以它产出的微 tile 集合——每个 `MR` 对齐的行偏移加上唯一那条 `m` 尾巴——在任何切分下都相同；更宽的 worker 数只是把同一批 tile 重新分组成更多、更小的行块。而唯一塑造 tile 累加顺序的分块维度 `KC` 从不随线程数移动。于是固定的输入、环境和配置在任何 worker 数下仍给出完全相同的输出——可复现契约照旧成立，连同 `MC` 的缩小一起。并行度在别处影响的是打包*决策*——LHS 打包闸门看的是每 worker 的列复用，共享 `A` 预打包只在大型并行问题上启用——但那些只决定打包字节放在哪里、由谁写入，从不改变计算出的数值。任务列表如何切分、契约如何端到端成立，见[并行执行](并行执行.md)；本页提到的旋钮以及所有其他 `GEMMKIT_*` 阈值，见[调优旋钮](../gemmkit-guide/调优旋钮.md)。
