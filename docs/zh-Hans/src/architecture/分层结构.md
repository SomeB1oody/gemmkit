# 分层结构

核心 crate 的每个模块开篇都会声明自己在栈中的位置：`api.rs` 写着 "Public core API (layer L8a)"，`driver.rs` 写着 "The generic GEMM driver (layer L4)"，一路排到 L0 的 `simd.rs`。这些标签不是装饰——它们是把 crate 的依赖纪律写在你不可能错过的地方。本页自底向上走一遍这个栈，这样等走到公共 API 时，它用到的每个词都已经有了定义。下一页[一次 GEMM 调用的生命周期](一次GEMM调用的生命周期.md)会沿相反方向再穿越一次，跟随一次真实调用。

```
L8a  api        安全切片入口、*_with、*_unchecked；MatRef/MatMut
L7   dispatch   运行时 ISA 选择，每个类型一个记忆化的函数指针
L6   special    gemv、small-k、small-m,n、批量的改道路径
L5   parallel   工作线程数决策、JobCursor 工作分发
L4   driver     泛型 5 层循环分块 GEMM，所有家族共用一个
L3   cache      拓扑探测 + BLIS 解析式分块
L2   pack       微面板打包原语
L1   kernel     KernelFamily 接缝（float/mixed/int/complex）+ Epilogue
L0   simd       ISA 令牌 + SimdOps 词汇表；scalar：Scalar/Acc 类型
     ---        横切模块：tuning（GEMMKIT_* 旋钮）、workspace（缓冲区）
```

## L0：词汇表——`scalar.rs` 与 `simd.rs`

最底层定义了 crate 其余部分被允许谈论的东西。`gemmkit/src/scalar.rs` 是数据类型接缝，而且刻意做得极小：

```rust
pub trait Scalar: Copy + Send + Sync + PartialEq + 'static {
    /// The type in which products are accumulated. `Self` for `f32`/`f64`
    type Acc: Scalar<Acc = Self::Acc>;
    /// The additive identity
    const ZERO: Self;
    /// The multiplicative identity
    const ONE: Self;
}
```

这就是整个 trait：单位元常量加累加器类型（`f16`/`bf16` 在 `f32` 中累加，`i8` 在 `i32` 中累加，`f32`/`f64`/复数在自身中累加）。它上面没有任何算术——真正的运算要么向量化地发生在 `SimdOps` 里，要么在各家族的标量 epilogue 里——所以新增元素类型永远不会连带出一套标量算术接口。细化 trait `Float`、`NarrowFloat`、`ComplexFloat` 按需叠加个别路径需要的额外能力。`scalar.rs` 刻意不知道的事：SIMD 的存在。它完全不知道自己的常量将来会被广播进向量寄存器。

`gemmkit/src/simd.rs` 加上 `simd/` 里的各后端（`avx512.rs`、`fma.rs`、`neon.rs`、`scalar.rs`、`wasm.rs`，以及 `complex.rs` 里的复数胶水）构成承重墙。两个 trait 分担职责：`Simd` 是零大小的 ISA **令牌**（`Avx512`、`Fma`、`Neon`、`ScalarTok`、`Simd128`，外加具备点积能力的 `Avx512Vnni`/`Avx512Bf16`），唯一的方法 `vectorize` 是 `#[target_feature]` 跳板，把运行时选中的 intrinsic 放进特性已启用的代码生成上下文；`SimdOps<T>` 是每个元素类型的厚词汇表——寄存器类型、`LANES`、load/store/broadcast/mul/add/fma/reduce，以及可覆写的 `accumulate_tile` 调度。再往上的 `KernelSimd<L, R, A, O>` 是拓宽/收窄接缝，让混合精度不需要驱动层的任何分支。这个模块刻意不知道的事：它上面的一切。模块文档写明它只依赖 `scalar` 和 `core`，因此可以原封不动拆成独立 crate——`SimdOps` 不知道什么是微面板、什么是缓存、什么是 GEMM。

## L1：运算家族接缝——`kernel.rs`

`gemmkit/src/kernel.rs` 与 `kernel/`（`float.rs`、`mixed.rs`、`int.rs`、`complex.rs`、`epilogue.rs`）定义 `KernelFamily`：把区分一种 GEMM 与另一种 GEMM 的所有东西打成一捆——`Lhs`/`Rhs`/`Acc`/`Out` 类型、打包布局（`pack_lhs`/`pack_rhs`）、微内核（`microkernel_epi`），以及告诉驱动如何为该家族分块的常量 `OUT_IS_ACC`、`DEPTH_MULTIPLE` 等。`FloatGemm<T>` 是基线；`MixedGemm`、`IntGemm`/`IntGemmVnni`、`ComplexGemm` 是原样复用驱动的兄弟家族。这一层还拥有带零成本 `Identity` 的 `Epilogue` trait，以及驱动预先算好、让微内核永远不必比较浮点数的 `AlphaStatus`/`BetaStatus` 枚举。家族刻意不知道的事：自己的 tile 尺寸。`MR_REG` 和 `NR` 是微内核方法上的 const 泛型，由高三层的分发点按 (类型, ISA) 选定——家族对任何几何都能编译，新 tile 是一次新的实例化，永远不是新类型。

## L2：机械拷贝——`pack.rs`

`gemmkit/src/pack.rs` 存放各家族打包钩子共同委托到的两个共享原语（唯一的例外是复数家族的实虚平面拆分打包，它随家族本身存放）：`pack_panels`，微面板为主序的拷贝（LHS 面板高 `mr` 行，RHS 面板宽 `nr` 列——同一个例程交换“领先”与“深度”步长即可通用，尾部补零，带步长的源走缓存分块转置），以及点积家族使用的 k 组交错变体 `pack_kgroup_panels`。它刻意不知道的事：自己的输出去了哪里。同一个例程既填充一次性调用的临时区，也填充并行共享的打包缓冲，还填充调用方长期持有的 `PackedRhs`——`pack.rs` 从未见过 `Workspace`、工作线程或生命周期，只见过 `dst`、`src` 和步长。正是这份漠不关心让预打包路径与逐调用路径的字节完全一致。

## L3：机器模型——`cache.rs`

`gemmkit/src/cache.rs` 及其后端（`cache/cpuid.rs`、`cache/sysfs.rs`、`cache/sysctl.rs`）回答两个问题：缓存层级长什么样，以及由此推出什么分块。探测是一条不可能失败的尽力而为回退链——x86 上 CPUID，然后 Linux sysfs，然后 macOS sysctl，最后是在 Zen5 上校准的静态默认值——`#[cfg]` 只用来挑选**探测方法**，从不直接给值，结果在 `Machine` 中记忆化一次。`blocking()` 随后按 BLIS 模型从缓存几何解析式地计算 `(MC, KC, NC)`：`KC` 让 A、B 微面板在 L1 中共存，`MC` 让 A 宏面板装进 L2，`NC` 让 B 宏面板装进 L3。关键类型是 `Level`（带着文档详尽的 `shared_by` 争用字段）、`CacheTopology` 和分块结果。这一层刻意不知道的事：线程数。`blocking()` 没有工作线程参数，而这个缺席是承重的——与线程数无关的分块正是[设计目标](设计目标与总体图景.md)里可复现性契约背后的机制。详见[分块与缓存模型](分块与缓存模型.md)。

## L4：引擎——`driver.rs`

`gemmkit/src/driver.rs` 是服务所有家族的那一个分块循环嵌套：BLIS 顺序的 `jc -> pc -> 扁平作业列表` 结构、自适应打包决策（B 按深度切片打包或就地读取，A 按工作线程打包、走共享预打包，或干脆不打包）、预打包 RHS 的消费路径，以及预打包 API 复用的 `pack_rhs_full` 布局——预打包 GEMM 和普通 GEMM 由此产生完全相同的面板字节。它的公开入口是 `run`、`run_epilogue`、`run_packed_rhs`、`run_packed_rhs_epilogue`，全部汇入私有的 `run_inner`。它刻意不知道的事：任何具体元素类型或 ISA。整个文件对 `Fam: KernelFamily` 和一个 `KernelSimd` 令牌泛型，从不点名 `f32`，从不点名 AVX-512，从不按元素类型分支。这就是开放封闭性质——新增家族或 ISA 不动这个文件一个字，`gemmkit/tests/open_closed.rs` 用一个 crate 并不发布的第二平凡家族驱动驱动器来证明这一点。

## L5：工作分发——`parallel.rs`

`gemmkit/src/parallel.rs` 拥有 `Parallelism` 枚举（`Serial` 或 `Rayon(n)`，`Rayon(0)` 为自动）、感知工作量的线程数决策（总工作量低于门限时保持串行；显式线程数被尊重但有上限；自动档随总工作量 `m*n*k` 扩展而不是直接跳到全部核心；gemv 形状另有一条带宽规则），以及按需分发的机械装置：`JobCursor`——工作线程从中拉取连续区块的无锁原子游标——加上 `job_grain`/`packed_block_grain` 的区块粒度计算和驱动当作屏障使用的 `for_each_worker` fork-join。它还提供 `Ptr`，让裸指针得以进入 rayon 闭包的 `Send + Sync` 垫片。它刻意不知道的事：作业是什么。`JobCursor` 对一个抽象计数发放索引区间——这个文件里没有任何地方提到 tile、矩阵或家族，所以同一个游标既调度驱动的 tile，也调度 B 打包面板、A 打包行块和 gemv 的行面板。详见[并行执行](并行执行.md)。

## L6：改道路径——`special.rs`

`gemmkit/src/special.rs` 与 `special/`（`gemv.rs`、`small_k.rs`、`small_mn.rs`、`batched.rs`）存放寄存器分块驱动并不适合的形状的专用路径：矩阵乘向量、低深度 GEMM、小 `m,n` 长 `k` 的内积形状，以及批量编排层。它们都藏在同样的公共入口背后，[特殊路径](特殊路径.md)有完整介绍。特殊路径刻意不知道的事：自己为什么被选中。门限——`gemv_threshold`、`small_k_threshold`、`small_mn_dim`——住在上一层的分发逻辑和旁边的 tuning 模块里；`small_k::run` 甚至分辨不出自己在服务 `gemm`、`gemm_fused` 还是 `gemm_map`，因为 epilogue 是作为不透明的泛型参数到达的。

## L7：运行时 ISA 选择——`dispatch.rs`

`gemmkit/src/dispatch.rs` 与 `dispatch/`（`isa.rs`、`float.rs`、`mixed.rs`、`int.rs`、`complex.rs`）把“这台机器该跑哪个内核”变成一锤子买卖。每个元素类型有一个 `OnceLock<Dispatched<T>>` 槽：特性探测只跑一次，胜出的单态化入口（普通、预打包、融合）连同 tile 几何被缓存，之后每次调用都是经由**带类型**函数指针的一次普通间接调用——没有 `transmute`，没有 `AtomicPtr<()>`。这一层还拥有问题描述符 `Task<T>`、`execute` 里的退化情形处理、方向归一化 `orient_transpose`、特殊路径门限，以及强制指定（或响亮失败于）特定内核的 `GEMMKIT_REQUIRE_ISA`。它刻意不知道的事：`Task` 里的指针从哪来。受检的切片视图和 unchecked 的裸指针到这里长得一模一样；校验发生在上面，或者根本没发生，分发层既不知道也不关心。参见 [SIMD 令牌与 ISA 分发](SIMD令牌与ISA分发.md)和面向用户的[运行时 ISA 分发](../gemmkit-guide/运行时ISA分发.md)。

## L8a：公共边界——`api.rs`

`gemmkit/src/api.rs` 与 `api/`（`batched.rs`、`cplx.rs`、`fused.rs`、`int8.rs`、`map.rs`、`packed.rs`）定义带步长视图 `MatRef`/`MatMut`、各家族的安全入口及其 `*_with`（调用方自备工作区）和 `*_unchecked`（裸引擎）变体、`validate_gemm_views` 的 panic 清单，以及视图到 `Task` 的降解。它刻意不知道的事：分发层以下的一切。API 层看不到将要运行哪个 ISA、会选出什么分块、打包会不会发生——校验完成后它把 `Task` 递给 `dispatch::execute`，任务就结束了。对称地，`MatRef` 从不出现在这一层之下；crate 的其余部分只讲指针和步长。

## 为什么箭头只指向下方

依赖方向是这套架构唯一的硬规则：每一层由上层驱动，且对上层一无所知。`simd` 只依赖 `scalar` 和 `core`；驱动从不点名元素类型或 ISA；L7 以下没人知道分发层的存在；L8a 以下没人听说过切片。三个收益支撑这份纪律。其一，扩展成本：因为知识只向下流动，新 ISA、新元素类型、新家族都在自己那一层接入，其下的一切可证明地原封不动——[扩展点](扩展点.md)描述的各条接缝之所以成立，正是因为没有任何下层可能对上层做过特判。其二，审阅局部性：审计微内核只需读 `kernel/float.rs` 和 `SimdOps` 契约，别无其他；审计调度只需读 `driver.rs` 和 `parallel.rs`。其三，可测试性：下层可以被隔离地检验（SIMD 一致性测试把每个令牌与标量模型比对；开放封闭测试用外来家族驱动驱动器），这正是[测试与验证](测试与验证.md)里正确性故事得以成立的原因。

## 两个横切模块

有两个模块立在栈的旁边而不是栈里，因为每一层都需要它们，而它们除 `core`/`alloc` 外不依赖任何上层。

`gemmkit/src/tuning.rs` 是统一的旋钮表面。引擎里每个启发式阈值——串行/并行门限、打包门限与步长、特殊路径阈值、调度粒度、分块上限——都住在这里，并按 **按调用参数 > 程序化 setter（`tuning::set_*`）> 环境变量（`GEMMKIT_*`）> 编译期默认值** 的优先级解析。环境变量只读一次并缓存；无法解析的值在 stderr 上告警后回退而不是 panic，因为性能旋钮里的笔误绝不应弄崩进程。全部 `GEMMKIT_*` 名字在 `tuning::knob_env_names` 注册表中枚举一次，crate 外的消费者（gemmkit-tune 的扫描表、旋钮属性测试、fuzz 的 setter 列表）都与它断言比对，因此新旋钮不可能悄悄逃出覆盖范围。面向用户的导览见[调优旋钮](../gemmkit-guide/调优旋钮.md)。

`gemmkit/src/workspace.rs` 是临时内存的故事。`Workspace` 是可增长、64 字节对齐的缓冲区；`Workspace::regions` 把它切成每工作线程（或每行块）的 LHS 区域加一个共享的 RHS 区域，并在元素数到字节数的必经关口做“失败即封闭”的溢出检查。在 `std` 下，一个可重入安全的线程本地池提供默认工作区，普通 `gemm` 每线程至多分配一次；`*_with` 入口改为穿入调用方自备的工作区，从第一次足够大的调用之后堆分配为零；没有 `std` 时每次调用使用全新工作区。详见[打包与工作区](打包与工作区.md)。
