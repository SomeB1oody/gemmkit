# 复数拆分内核

复数 GEMM 是唯一不搭乘 `FloatGemm` 的同质类型家族。理论上它可以：`Complex<f32>` 和 `Complex<f64>` 都在自身类型中累加（`Lhs = Rhs = Acc = Out`），这正是浮点家族处理的形态。问题出在内存布局上。`num_complex` 把一个复数存成相邻的 `(re, im)` 对，从复数切片加载的 SIMD 寄存器因此持有 `re, im, re, im, ...`，而一次复数乘法需要跨通道组合：实部是 `re*re - im*im`，虚部是 `re*im + im*re`。在交错通道上，这些组合迫使最内层循环里出现 shuffle 和 `fmaddsub` 一类指令，每个深度步一次，总共 `O(mnk)` 次。gemmkit 的做法是在打包时把布局改写一次，让热循环保持纯实数 FMA，循环内没有任何 shuffle。本页依次讲这个拆分设计、共轭如何从中免费落出、内核经由的接缝、微块形状背后的寄存器预算算术，以及数值上的保证。

## 拆分布局

家族是 `gemmkit/src/kernel/complex.rs` 里的 `ComplexGemm<T, CONJ_A, CONJ_B>`，设计的核心在它的打包例程里。`pack_planar` 把每个微面板按结构数组（SoA）形式铺开：每个深度步先存 `width` 个实部、紧接着存 `width` 个虚部（LHS 的 `width` 是 `mr`，RHS 是 `nr`，步长互换的方式与共享的 `pack_panels` 完全一致）。内核于是用普通的连续加载取到一整寄存器实部和一整寄存器虚部，去交错的成本从 `kc` 内层循环移进打包：摊销后是 `O(MK + KN)` 而不是 `O(MNK)`。

因为内核只能消费这种平面布局，两个操作数永远都要打包：家族设置 `FORCE_PACK_LHS = FORCE_PACK_RHS = true`，压过驱动层基于代价的原地读取决策。`pack_planar` 复刻了 `pack_panels` 的两条写入路径——先导维连续时的顺序直拷，以及带步长源的缓存分块转置——所以行主序操作数打包时不会每个元素错过一次缓存；两条路径写出的面板逐字节相同，只是写入顺序不同（共享框架见[打包与工作区](打包与工作区.md)）。

## 共轭是打包时的符号翻转

共轭只对虚部取负，而打包本来就单独写虚部平面，所以 `conj(A)*B` 和 `A*conj(B)` 在热循环里零成本：`CONJ_A`/`CONJ_B` const 泛型被设置时，打包器在拷贝时对虚部平面取负（真正的取负，`+0.0` 映射到 `-0.0`，与 `num_complex` 的 `.conj()` 一致），同一个实数 FMA 循环原样运行，任何地方都没有逐元素的共轭分支。这也是强制打包标志的第二个理由：当打包不止是普通拷贝时，变换必须保证执行。

运行时到编译时的桥梁在 `gemmkit/src/dispatch/complex.rs`。公开入口 [`gemm_cplx`](https://docs.rs/gemmkit) 以普通 `bool` 接受 `conj_a`/`conj_b`；`run_complex` 对这对布尔值只 match 一次，分发到四个 `ComplexGemm` 单态化中匹配的那个，所以分支每次调用只发生一次，从不进入循环。有一处细节随之而来：把行主序倾向的 `C` 规范化的方向交换实际计算 `C^T = B^T * A^T`，而 `(conj(A)*B)^T = B^T * conj(A)^T`，所以交换时必须连同共轭标志一起交换——match 之前就完成了。输出共轭（`conjC`）未实现。在退化路径（`k == 0` 或 `alpha == 0`）上标志干脆无关紧要：没有 `A*B` 项，也就没有什么可共轭的。

## 热循环：每次复数乘加四条实数 FMA

循环能跑起来之前还有一个分层问题要解决。家族是同质的，驱动层的约束是 `T` 为复数的 `KernelSimd<T, T, T, T>`，它只提供 `SimdOps<Complex<..>>`，而不是拆分内核真正需要的实数运算。桥梁是 `SimdOps::cplx_microkernel` 接缝：家族的 `microkernel` 转发给它，每个 ISA 令牌的覆写（由 `gemmkit/src/simd/complex.rs` 的胶水宏 `impl_complex_simd!` 生成）再转发给唯一共享的、ISA 泛型的 `soa_microkernel`——它写在 `S: SimdOps<C::Real>` 之上。累加器在家族接缝处保持复数类型，复数的 `alpha`/`beta` 原样穿过驱动层；接缝之内，累加器是两组实数寄存器。薄薄的 `SimdOps<Complex<..>>` 胶水只是为了让驱动层能读 `LANES`、让同质 blanket impl 成立；它的元素运算全是 `unreachable!`（复数 GEMM 从不调用它们），而 `LANES` 设为实数通道数，一个实数通道对应一个复数行，驱动层的 `mr = MR_REG * LANES` 数的就是复数行。

循环本身，摘自 `gemmkit/src/simd/complex.rs`：

```rust
for p in 0..kc {
    let are_p = a_re.add(p * 2 * mr); // re plane of this depth step
    let aim_p = are_p.add(mr); // im plane (offset by `mr`)
    let ar: [<S as SimdOps<C::Real>>::Reg; MR_REG] =
        core::array::from_fn(|i| simd.loadu(are_p.add(i * lanes)));
    let ai: [<S as SimdOps<C::Real>>::Reg; MR_REG] =
        core::array::from_fn(|i| simd.loadu(aim_p.add(i * lanes)));
    let bre_p = b_re.add(p * 2 * NR);
    let bim_p = bre_p.add(NR);
    for j in 0..NR {
        let br = simd.splat(*bre_p.add(j));
        let bi = simd.splat(*bim_p.add(j));
        for i in 0..MR_REG {
            acc_re[j][i] = simd.mul_add(ar[i], br, acc_re[j][i]); // += ar*br
            acc_re[j][i] = simd.fnma(ai[i], bi, acc_re[j][i]); //    -= ai*bi
            acc_im[j][i] = simd.mul_add(ar[i], bi, acc_im[j][i]); // += ar*bi
            acc_im[j][i] = simd.mul_add(ai[i], br, acc_im[j][i]); // += ai*br
        }
    }
}
```

一次复数乘加是流入两组累加器的四条融合实数步：一条 `mul_add` 加一条 `fnma`（融合取负乘加，x86 上是 `vfnmadd`）进 `acc_re`，两条 `mul_add` 进 `acc_im`。每条操作都是作用在连续加载和标量广播上的普通逐通道 FMA；在 epilogue 之前没有任何跨通道操作。固定的逐 `p` 顺序是刻意的：正是它让同一矩阵的完整微块与边缘微块舍入完全一致。

循环之后，两组累加器排入平面暂存区，由标量 epilogue 折叠复数 `alpha`（`alpha == 1` 时跳过复数乘法）、按情形合并 `beta*C`、在写出时重新交错——一次摊销 `O(MN)` 的扫描，统一处理完整、边缘和带步长的输出微块。打包中的标量去交错和 epilogue 中的标量重交错是实测后的决定而非疏忽：在 NEON 上试过按 ISA 的 `vld2`/`vst2` 向量化接缝，没有收益——内层循环占绝对主导，通用标量路径就是下限。

## 寄存器压力与 NR 的选择

拆分设计让累加器数量翻倍：一个 `MR_REG x NR` 的复数微块需要 `2*MR_REG*NR` 个累加寄存器（实部一组、虚部一组），外加 `2*MR_REG` 个 A 平面寄存器和每列步 2 个 B 广播。这份预算在 `gemmkit/src/dispatch/complex.rs` 里逐微块记录在案，它让复数微块比浮点的小，本页唯一一个来之不易的数字也源于此。在 FMA（16 个 YMM 寄存器）上，c32 取 `MR_REG = 1`（8 个实数通道即 8 个复数行）、`NR = 5`：10 个累加器 + 2 个 A + 2 个 B 广播 = 16 个寄存器中占 14 个。空出的两个至关重要：`NR = 6` 的 16 占 16 满配微块会把累加器溢出到栈上，吞吐大约减半，所以 `NR` 收缩到 5，代码注释记下了原因。AVX-512 的 32 个 ZMM 缓解了压力：c32 取 `MR_REG = 2`、`NR = 6`，用掉 24 + 4 + 2 = 32 中的 30。NEON（32 个向量寄存器）取 `MR_REG = 2`、`NR = 5`，占 32 中的 26，给在途加载临时量留出空间；wasm `simd128` 取 `MR_REG = 1`、`NR = 4`，共 12 个 v128。各个 c64 变体沿用其 c32 兄弟的 `MR_REG`/`NR`、通道数减半——预算算术与通道数无关。

## 精度与可复现性

复数没有特殊路径：`run_complex` 把所有形状都送进 `driver::run`，没有 gemv、`small_mn` 或小 `k` 分支（[特殊路径](特殊路径.md)那套机制只服务实数浮点和整数）。这让数值契约很好陈述，`gemmkit/tests/correctness/complex.rs` 对每一条都有直接测试。

确定性与线程无关性按位成立：分块与线程数无关，同一问题的串行与并行运行产生逐位相同的输出，测试直接断言各线程数下 `re`/`im` 原始位模式相等。在单次运行内部，四条 FMA 的固定逐步顺序意味着同一矩阵的完整微块和边缘微块舍入一致，结果不依赖微块边界恰好落在哪里。共轭完全不引入舍入——它是对精确值的符号翻转——一个专门的小整数输入测试（其中每个乘积与和都精确可表示）用精确相等而非容差，对照朴素参考检验全部四种共轭组合。

对外部 oracle 的标准必然更宽：正确性套件把 `gemm_cplx`（含全部共轭组合）与 `gemm` crate 在套件惯用的 L2 型容差下对比（另有针对负行步长视图加共轭的测试，对照行反转的参考），因为分块的 SoA 收缩与另一引擎的求和顺序舍入不同是正当的。这正是[可复现性契约](设计目标与总体图景.md)在复数上的体现：相同的输入、环境与配置给出相同的位；不同引擎或不同顺序按容差一致。

融合偏置入口是拿到按位保证的例外。`gemm_cplx_fused` 支持逐行或逐列的复数偏置，并且刻意不提供激活——ReLU 一类基于序的激活在无序域上没有定义。它的 epilogue 从不触碰内核的算术：SoA 内核存入的正是普通 `gemm_cplx` 会存的那些位，一个微块局部的后处理只在最后一个深度面板上就地映射它们（复数是 `OUT_IS_ACC = true`，中间面板必须保留原始部分和）。结果与先跑 `gemm_cplx` 再做同样的逐元素偏置加法逐位相同，对每种形状、每种共轭组合都成立；`Identity` 单态化把后处理整个常量折叠掉，非融合路径不为这个钩子的存在付任何代价（通用机制见 [Epilogue 融合](Epilogue融合.md)）。
