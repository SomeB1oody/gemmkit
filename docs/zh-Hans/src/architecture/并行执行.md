# 并行执行

gemmkit 的并行是一个小小的层（`gemmkit/src/parallel.rs`，L5），只做两件事：决定一个问题*值得*多少个 worker，以及把*哪些*工作交到每个 worker 手里。两个决定都刻意保守——多开线程并不免费，这一层的设计出发点就是：错误的 worker 数比错误的调度损失更多性能——而且两者的形状都保证数值结果不依赖其中任何一个。面向用户的界面只有一个枚举：`Parallelism::Serial`，或 `Parallelism::Rayon(n)`，其中 `Rayon(0)`（默认值）表示自动。

## 感知负载的 worker 数解析

`Parallelism::resolve` 把请求变成实际的分区数，而请求只是输入之一，负载是另一个。首先是总工作量的串行闸门：`m*n*k` 低于 `GEMMKIT_PARALLEL_THRESHOLD`（默认 `48*48*256`）时一切保持串行，连核数都不采样——为一个微秒级的乘积去 fork rayon，代价比乘积本身还高。闸门先于请求生效，所以低于它时连显式的 `Rayon(n)` 也保持串行。过了闸门，显式计数被尊重，但以核数和可用任务数封顶，因此 `Rayon(huge)` 既不能过度订阅机器，也不能多分配按 worker 的打包区域——只有自动路径是启发式的，这让强制宽度对测试和扩展性诊断保持精确。

自动路径以工作量为准。它把总工作量 `m*n*k` 除以 `GEMMKIT_PAR_MNK_PER_WORKER`（默认 `2_000_000`，即实测的每 worker 下限，低于它 fork/join 开销会吃掉收益）得到 worker 数，下限为一，并以核数和任务数封顶。之所以按工作量而非维度计，是因为实测最优值跟随的是总 flops 而不是线性尺寸：在 Zen5 9950X 上，`128^3` 串行最快，`192^3` 想要两三个 worker，而 `384^3` 已经想用满全部 32 个硬件线程——这样的跨度没有任何单一的线性维度步幅能拟合。早先的设计改用 `cbrt(m*n*k)` 除以一个从核数推导的步幅来渐升，理由是中等规模 GEMM 的并行度足以*占据*每个核，却没有足够的单 worker 工作量去*摊销*它们，所以开满全宽会输给 fork/join 和争用。可后来发现，那个中等规模的凹陷是各 worker 冗余打包 LHS 的假象——每个 worker 都在重打同一条 `A` 面板，多出来的 worker 付出的主要是打包流量——而不是 worker 数本身之过。一旦 LHS 就地读取门（[打包与工作区](打包与工作区.md)里的步长/跨度这一对）止住了那份冗余打包，中等规模便直接扩展到全宽：自动 `512^3` 从 8 个 worker、1329 GFLOP/s 变成 32 个 worker、2752 GFLOP/s。于是坡道被退役，换成这条扁平的、以工作量为准的计数，而环境变量旋钮是留给每 worker 下限与 9950X 不同的机器的逃生口。

带宽受限的形状适用一条完全不同的规则。gemv 或 gevv 每字节只做 O(1) 次算术，计算坡道的逻辑不成立：`resolve_bandwidth` 改按*触碰的字节数*设闸。低于一个从末级缓存推导的字节下限（`cache.rs` 的 `gemv_parallel_floor_bytes`：有 L3 时取每核 L3 份额的四分之一，无 L3 的机器取整个簇共享 L2 的一半；`GEMMKIT_GEMV_PARALLEL_BYTES` 可覆盖）时，触碰的数据驻留在缓存里，单核已经拿到全部末级缓存带宽——拆分只会增加 fork/join 和共享缓存争用，却没有 DRAM 带宽可赚。高于下限时，自动计数*直接*跳到带宽上限（x86 取逻辑核数的四分之一——为共享载入/存储单元的 SMT 兄弟先除以二，再因为一部分物理核就能打满 DRAM 又除以二；无 SMT 的 aarch64 取一半；下限为 2；`GEMMKIT_GEMV_THREAD_CAP` 可覆盖）。不设坡道，因为带宽扩展曲线上*少量 worker 恰是最糟的点*：曲线在那里下凹——fork/join 和争用的成本已经付了，聚合的 DRAM 带宽还没到手——任何穿过凹陷的坡道都输给两个端点。下限以下串行，以上直达上限，中间什么都没有。

批量 GEMM 有自己的解析器 `resolve_batch`，在三种计划中选择：`Serial`；`BatchParallel(n)`——每个 worker 串行地跑完整的、缓存热的 GEMM，整批只付一次 fork/join，且因为没有元素被拆开，在任何 worker 数下逐位一致；以及 `SequentialInternal`——顺序遍历批次，轮流让每个大型、DRAM 受限的元素独享整个引擎的并行度。拆分计划只对 `m, n > 1` 的形状开放，它们的路由与 worker 数无关；gemv 形状的元素永远整体留在一个 worker 上。路由细节见[特殊路径](特殊路径.md)，API 见[批量GEMM](../gemmkit-guide/批量GEMM.md)。

## 按需拉取的工作分发

定下 worker 数后，驱动器并不搭建嵌套任务树。对每个列块和每个深度片，它把内部工作压平成一维任务列表——`n_mc` 个行块乘 `n_nt` 个列瓦片，任务 `q` 解码为 `(ic_idx, jt) = (q / n_nt, q % n_nt)`——worker 从一个共享的无锁游标里拉取连续的块，直到取空：

```rust
// gemmkit/src/parallel.rs
impl JobCursor {
    /// Atomically claim the next `[start, end)` chunk, or `None` once the job space
    /// is exhausted
    #[inline]
    pub(crate) fn next_chunk(&self) -> Option<(usize, usize)> {
        let start = self.next.fetch_add(self.grain, Ordering::Relaxed);
        if start >= self.n_jobs {
            None
        } else {
            Some((start, (start + self.grain).min(self.n_jobs)))
        }
    }
}
```

每次认领一条 `fetch_add`，没有锁，没有按任务的队列。按需拉取正是异构核心跑得好的原因：在大小核机器上，块干得快的 P 核自然按比例多拉几块，而静态的 `n_jobs / n_threads` 均分会让所有核等最慢的那个。同一性质也吸收了同构机器上的系统噪声和频率差异。

块粒度在两种成本间权衡。太粗，任务列表的尾巴让 worker 在汇合点闲着；太细，原子认领（以及打包 LHS 路径上块边缘的重复打包）的开销开始显形。通用粒度对 worker 数过采样：`job_grain` 以每 worker `GEMMKIT_PARALLEL_OVERSAMPLE` 块为目标（默认 8），每个 worker 预期拉取多次，失衡便能自我修正。打包 LHS 路径是特例：它的天然块是整个行块（`n_nt` 个连续任务），worker 把该行块的 `A` 面板打包一次，在这个行块的所有列瓦片上复用。这样只有 `n_mc` 个块，行块数少时，`packed_block_grain` 把每个行块按 2 的幂切成列子块，直到大约 `GEMMKIT_PACKED_OVERSAMPLE * n_threads` 个块（默认目标 2）——但只用 `n_nt` 的因数切，块因此永不跨越行块边界、不会在块中途重打 `A`。这个切分目标是扫描得出的最优点：切得更狠会过于频繁地重打包，反而退步。

每个深度片里有两个并行阶段先于计算区域，它们的边界是驱动器里仅有的屏障。`B` 打包时，worker 从各自的游标拉取 `nr` 宽的列面板，该区域的 fork/join 就是计算区域依赖的先写后读屏障——打包后的 `B` 是唯一被所有 worker 非互斥共享的缓冲区。共享 `A` 预打包（越过尺寸门槛即开启，且从 16 个 worker 起无论尺寸都开启——宽度到了那里，每多一个 worker 就是它所触及面板的又一份冗余拷贝，去重在中等规模上也已然划算）按同一纪律把每个行块的面板一次性打进共享槽位。其余一切按构造互斥：worker 只写自己的输出块和自己的打包区域，这条不变量让 `Ptr` 垫片得以在唯一一处经过审计的地方把捕获的裸指针声明为 `Send + Sync`。

## 按规模分级的线程池

rayon 的 fork/join 代价并不随问题规模而变，它随的是池的*空闲余量*——池所拥有的线程数与某次调用实际用到的 worker 数之间的差距。向一个 `w` 宽的池 fork `k` 个 worker，被唤醒的是 `w` 个线程而非 `k` 个，那多出来的 `w - k` 个虽然分不到活干，仍要为屏障、以及操作系统级的唤醒/挂起往返各付一份代价。对一次小规模并行 GEMM 而言，全宽的全局池是最糟的情形：在一台 32 线程的 Zen5 机器上，`256^3` 的乘积大约只想要 8 个 worker，于是每次把它 fork 进 32 宽的全局池，都要在 24 个纯属空闲的线程上白付一份空闲税。

gemmkit 的应对是一小撮私有的、持久化的池，尺寸恰好卡在自动路径实际会请求的 worker 数上。在满机器宽度之下，它最多保留 `GEMMKIT_POOL_CLASSES`（x86_64 上默认 2，封顶 3；其余架构默认 0，等待设备实测再开启）个减半档位——在一台 32 线程的机器上，即半宽（16，物理核数）与四分之一宽（8）。每个档位池都是惰性构建的：首次用到时才建，一次性代价约为每线程 12 微秒，此后再也不重建；这些档位是机器宽度的固定减半，而非按形状调优出的值。自动路径把它的 worker 数精确地卡在某个档位上——按构造即为零空闲——并停留在物理核档位，直到总工作量 `m*n*k` 越过 `GEMMKIT_FULL_WIDTH_MNK`（默认自动：在 9950X 上为 110_000_000），到那时 SMT 宽度那部分 worker 才终于赚回了它们所付的 fork/join 代价，全机宽度才会接管。

三条规则确保这套机制永远不会与调用方自己的调度打架。一次已经跑在某个 rayon 池里的调用——无论是调用方自己的 `install`，还是嵌套在另一次 gemmkit 调用里——都不会被改道到某个档位池：环境池始终胜出，和档位池出现之前完全一样。显式的 `Rayon(n)` 保持其精确语义——恰好 `n` 个 worker——只是会挑一个能容纳 `n` 的最小档位池来 fork，而不是 fork 进全局池；worker 数本身不受影响，变的只是它 fork 进哪一个屏障。线程化 wasm 保留自己专属的池（见下文），不受这一切影响——档位池是原生、非 wasm 的话题。

这些数字是在 9950X 上标定的：四分之一宽档位（8 线程）在 `128^3` 到 `256^3` 之间大致把吞吐翻倍（`256^3`：档位池 2060 对全局池 1062 GFLOP/s），半宽档位（16 线程，即物理核数）在直到 `448^3` 之前都以 7%-11% 的优势胜过全宽，而全宽则从 `512^3` 起反超——与上面约 110_000_000 的 `m*n*k` 门槛一致。

## wasm 的故事

`wasm32-wasip1` 上没有线程可开，rayon 一试就会 trap。编译期常量 `RAYON_USABLE` 记录目标究竟能不能跑 worker：未开启线程选项的 wasm 构建里，所有解析器都返回 1，`for_each_worker` 跑普通的串行循环——`parallel` 优雅降级而不是 trap。选项就是 `wasm_threads` feature，面向 `wasm32-wasip1-threads`（或带 SharedArrayBuffer 的浏览器）：wasm 上 `available_parallelism` 不可用，rayon 的全局池会悄悄把自己配成单线程，所以 gemmkit 自建一个池，大小由 `GEMMKIT_WASM_THREADS` 旋钮决定（默认 8），并把 worker 循环装进这个池里执行。部署者声明宽度，其余一切不变。wasm 构建的细节见 [no_std与WebAssembly](../gemmkit-guide/no_std与WebAssembly.md)。

## 拼装可复现契约

gemmkit 承诺：固定的输入、环境和配置，在任何 worker 数下产出完全相同的结果。至此，支撑这条承诺的三个机制都已登场；下面是它们如何拼在一起。

第一，数值与 worker 数无关。[分块](分块与缓存模型.md)只从缓存模型推导 `kc` 和 `nc`，从不看线程数，深度片也始终按同一个固定的 `pc` 顺序执行，所以每个输出元素的浮点归约在一个 worker 排空游标和三十二个 worker 排空游标时形状完全相同。worker 数唯一能挪动的是 `mc`：宽 worker 数会把它缩小（驱动器的并行作业深度下限），好让扁平任务列表在每个 worker 上仍保持几个块深，因此任务列表在不同宽度下未必逐字节一致。但 `mc` 始终是 `mr` 的倍数，所以微 tile 集合——每个 `mr` 对齐的行偏移加上唯一那条 `m` 尾巴——在任何切分下都相同，而 `kc` 与 `pc` 顺序原封不动，没有任何结果比特移动。worker 数改变的是工作*如何分组与分走*，从不改变每个 tile 算什么。

第二，任何归约都不被拆开。在一个深度片内，一个输出块的整次更新由一个 worker 完成——一个块就是一组完整的输出块——而深度片本身顺序执行（`pc` 循环从不并行），`beta` 只作用于第一个片，之后累加。于是每个输出元素的浮点归约都按一个仅由分块决定的固定顺序进行。特殊路径遵守同样的纪律：gemv 按寄存器面板边界分割输出*行*，每一行的 SIMD/标量切分与分区方式无关——这让 gemv 在任何 worker 数下干脆逐位一致——批量计划则要么把完整元素留在一个 worker 上，要么只拆分路由与 worker 数无关的形状。

第三，打包出的字节与打包者无关。`pack_panels` 是纯重排，两条分支写出的字节完全一致，所以由 3 号 worker 打的面板、由共享 `A` 预打包打的面板、或几周前 `prepack_rhs` 打的面板，都是同一串字节（[打包与工作区](打包与工作区.md)）；内核无从观察是谁备好了它的输入。

哪个 worker 算哪个块确实每次运行都不一样——游标把块交给先来的人——但由上述三个机制，没有任何数值依赖于此。当下，驱动器路径上的串行与并行运行事实上逐位相等，因为两者在同一调度上跑同一个内核；*承诺*的契约是稍弱一点的“固定配置下可复现”，这为 bf16 点积路径这类按容差保持的内核留出了空间。实践中如何选择 worker 数、契约对测试意味着什么，见[并行实践](../gemmkit-guide/并行实践.md)。
