# 打包与工作区

微内核对输入只认一种形状：每个深度步上，`mr` 个 `A` 元素在内存中连续、`nr` 个 `B` 元素在内存中连续，面板一条接一条，中间没有任何杂质。用户矩阵几乎从不长这样——行列步长任意，尾部除不尽微块尺寸，深度方向的遍历还可能大步跨越内存页。打包（packing）就是弥合这道缝隙的那次拷贝：把每个宏块重排成*微面板优先*（micropanel-major）布局，一次到位，让最内层循环从 64 字节对齐的暂存区里读到纯单位步长的数据流，每次都是完整的 `mr`/`nr` 向量。这次拷贝是 `O(mc*kc)`，而复用它的计算是 `O(mc*kc*nc)`，所以摊销得起——也正因如此，当复用低到摊不平时，驱动器会干脆跳过它。

## 一个例程，两个操作数

机械的拷贝集中在一个例程 `pack_panels`（`gemmkit/src/pack.rs`，L1 层）里，因为 LHS 和 RHS 的布局其实是同一种布局从两侧看的结果。LHS 宏块打包成 `mr` 行高的面板、按列逐列存放：面板 0 存第 `0..mr` 行，每个深度步的 `mr` 个元素连续排布，面板 1 存第 `mr..2*mr` 行，依此类推。RHS 宏块打包成 `nr` 列宽的面板、按行逐行存放。两者都是“每个深度步 `width` 个连续前导元素”，唯一的差别是哪条矩阵轴充当前导轴，于是两个 `KernelFamily` 钩子调用同一个例程，只是把步长对调：

```rust
// gemmkit/src/kernel/float.rs
#[inline]
unsafe fn pack_rhs(
    dst: *mut T,
    src: *const T,
    rs: isize,
    cs: isize,
    kc: usize,
    nc: usize,
    nr: usize,
) {
    // RHS panels are `nr` columns wide, stored row-by-row: the "leading"
    // direction is columns (stride `cs`) and the "depth" is rows (stride
    // `rs`), the transpose of the LHS case, handled by swapping strides
    unsafe {
        pack_panels(
            dst, src, /*lead*/ cs, /*depth*/ rs, /*n_lead*/ nc, kc, nr,
        )
    }
}
```

`pack_lhs` 是镜像：`lead = rs`、`depth = cs`、`width = mr`。块除不尽时，尾部面板的空车道用零填充，因此内核永远读完整的 `mr`/`nr` 向量，边缘块在乘法本身里不需要任何掩码。

例程内部有两条写出字节完全相同结果的路径。前导维连续时（`lead == 1`——列主序的 `A` 或行主序的 `B`），每个深度步的 `live` 个元素在源里本就相邻，面板就是一串直白的 `copy_nonoverlapping` 加尾部零填充。前导维带步长时，朴素的逐元素收集每个元素都可能吃一次缓存未命中（每个深度步 `width` 次跨步加载）；例程改跑一趟*缓存分块转置*：沿源的连续维以 `GEMMKIT_PACK_TRANSPOSE_TILE` 个深度步（默认 16）为一条，把每条散布进面板。这是纯粹的重排拷贝——打包出的字节完全一致——但对跨步源便宜得多，也正是行主序 `A` 布局代价不比列主序高多少的原因。点积家族（`i8` VNNI、`bf16` `vdpbf16ps`）有一个姊妹例程 `pack_kgroup_panels`，在此之上把每条车道的 `DEPTH_MULTIPLE` 个连续深度步交织到一起，让一条点积指令一次吞下整组；那种布局属于[点积内核与深K孪生](点积内核与深K孪生.md)。

打不打包由驱动器决定，而且两个操作数并不对称。微内核以 `mr` 宽向量读 `A`，所以只要 `A` 的行不是单位步长、或行面板不完整，`A` 就*必须*打包；除此之外，当每个 worker 的列复用超过 `GEMMKIT_LHS_PACK_THRESHOLD`（aarch64 默认 256 列，其余目标默认 1024），或者列主序 `A` 的深度遍历同时满足步长页级*且*跨度够宽*且*复用的列块足够多、值得付出这个成本时也会打包——即每步的步长达到半个内存页（`GEMMKIT_LHS_PACK_STRIDE`，从 `Machine` 记忆化的页大小自动推导），*并且*整条深度切片遍历（`csa * sizeof(Lhs) * kc`）达到 `GEMMKIT_LHS_PACK_SPAN` 字节（自动：4 MiB），*并且*至少有 `GEMMKIT_LHS_PACK_REUSE` 条 `nr` 宽列块复用每个打包面板（`min(n, nc) / nr`，向上取整；x86 默认 128，aarch64 默认 4）。三道门必须同时成立：跨度仍驻留在缓存内的页级步长只是重走温热的缓存行，就地读取比它本要付的打包更快（在 Zen5 9950X 上实测，`n = 1024` 的 2 MiB 跨度内就地胜出，`n = 2048` 的 4 MiB 跨度起打包胜出），所以跨度伴侣让 `A` 保持就地，直到遍历真的宽到不分复用高低都会打垮 TLB。复用门槛为同一笔交易的另一面定价：瘦高形状（`m` 远大于 `n`）只靠极少的列块就堆出巨大的跨度，把一次昂贵的打包摊到过少的复用上并不划算——同样在 Zen5 9950X 上实测，就地在 86 个列块的复用内胜出 18%-71%（瘦的 `n` 对 4096 到 8192 的 `m`），而 171 个列块复用的深 K 方阵（`2048^3`）仍以 8% 的优势想要打包，这就是为什么那里的门槛默认为 128 个列块。在 aarch64 上这笔账几乎反转——M4 Max 打包便宜、而就地遍历要跨 16 KiB 的页——打包从 4 个列块的复用起就获胜（`m = k = 4096`：`n = 32` +88%，`n = 256` +220%），因此 aarch64 的门槛是 4。相比之下 `B` 只以广播单元素的方式被读取，任何布局不打包都能用，打包纯粹是为了复用：每个深度片一次，条件是 `m` 超过 `GEMMKIT_RHS_PACK_THRESHOLD`（默认 2048）、足够多的行块会反复读它。这些打包由谁执行、打包与计算之间的屏障如何安排，是[并行执行](并行执行.md)里的调度话题。

## 预打包操作数

同一个操作数在一次次调用中反复出现时——固定权重对一串激活的推理模式——按调用打包就是白费功夫。`gemmkit/src/api/packed.rs` 里的预打包入口把整个操作数一次性提前打好：`prepack_rhs` 沿步长遍历任意布局的 `B`，返回一个 `PackedRhs<T>`；`gemm_packed_b` 随后对它做乘法，完全跳过按调用的 RHS 打包。这套 API 的使用侧见[预打包操作数](../gemmkit-guide/预打包操作数.md)；从架构上看，有三条性质要紧。

第一，缓冲区记录了它构建时的分块几何——`nr`、`kc`、`nc`——消费调用原封不动地读回：驱动器用记录的 `kc`/`nc` 顶替自己模型的输出（只有 `mc` 仍按真实的 `m` 推导），因此面板地址永远与打包时一致，哪怕打包与消费之间有调优旋钮变了。几何本身通过与普通调用相同的 `blocking` 模型求解，行数用 `tiny_block_dim() + 1` 的哨兵值，从而绕开小矩阵分支，与最终的 `m` 无关。

第二，布局只有一个事实来源：`prepack_rhs` 通过 `driver::pack_rhs_full` 填充缓冲区，其面板铺设顺序与驱动器自己的按片打包写出的完全一致——最外层是 `jc` 块，然后是深度片，然后是每片的 `nr` 宽面板。预打包的字节等于按调用打包的字节，所以同一配置下预打包 GEMM *复现*普通 `gemm`；文档标注的例外是极小问题（`m` 和 `n` 都不超过 `tiny_block_dim`）和 gemv 形状——普通 `gemm` 会把它们改道到特殊路径，因而可能在最后一个 ULP 上有差。第三，缓冲区在 GEMM 期间只读，所以所有 worker 无同步地共享它——与按调用的 `B` 打包不同，它不需要任何屏障。

`PackedLhs` 几乎不花额外代码，靠的是引擎的 A/B 对称性：一个 `m x k` 的 LHS *就是*转置问题 `C^T = B^T*A^T` 的 RHS，所以 `prepack_lhs` 把步长对调后委托给 `prepack_rhs_unchecked`，`gemm_packed_a` 则经由转置问题消费。这个对称性也解释了取向断言：预打包 `B` 要求近列主序的 `C`（`|csc| >= |rsc|`），预打包 `A` 要求近行主序的 `C`——另一种取向会让分发层交换操作数角色，烘焙好的布局就派不上用场了。

`int8` feature 增加了异构孪生 `prepack_rhs_i8` / `gemm_i8_packed_b`，有三处刻意的不同。它的布局钉死在本进程记忆化分发所选中的那个整数内核上——VNNI 的 k 四元交织布局，或加宽内核的普通面板——消费入口永远运行同一个家族，缓冲区因此不可能被读错。它把缓冲区深度向上取整到点积内核的 `DEPTH_MULTIPLE = 4`，并把整个收缩打成单个深度片，满足驱动器对深度补齐家族的单片守卫。它还刻意绕过普通 `gemm_i8` 在 `GEMMKIT_I8_VNNI_MIN_PAR_MNK` 以下应用的小并行加宽回退：`vpdpbusd` 的缓冲区是四元交织的，加宽内核根本无法消费；而整数累加是精确的，无论走哪条路结果都与普通 `gemm_i8` 逐位一致。预打包恰恰在这条路径上收益最大——VNNI 的 RHS 打包本来每次调用都是强制的，小 `m` 时按调用的 `O(k*n)` 打包会压过 `O(m*k*n)` 的计算。

## 工作区

这一切打包都需要暂存内存，`Workspace`（`gemmkit/src/workspace.rs`）就是它的分配器：一块可增长、64 字节对齐（足够 AVX-512 存储）的缓冲区，按 2 的幂增长、从不收缩。每次调用中，`Workspace::regions` 把它切成 `a_regions` 个等大的 LHS 区域加一个共享的 RHS 区域，每个区域向上取整到对齐边界。LHS 区域数在按 worker 打包路径上等于 worker 数，在共享 `A` 路径上等于行块数——切法完全相同——当两个操作数都不打包时，驱动器干脆跳过预留，全原地的负载永远不会撑大池子。

### 在字节乘积处失败即拒绝

尺寸运算里藏着一个内存安全的微妙点。gemmkit 接受广播（零步长）视图：它们用一小片后备存储就能通过边界校验，却呈现出逼近 `isize::MAX` 的*逻辑*维度——于是给打包缓冲区计尺寸的乘积真的可能溢出 `usize`，而一个回绕（偏小）的尺寸会让缓冲区分配不足，随后的打包越界写入。驱动器用 `checked_mul` 守住了元素数乘积，但只查元素数不够：以混合精度路径上的 `k = 2^56` 为例，那里 `kc == k`。一个 `mc * kc` 元素的 LHS 区域——比如 `32 * 2^56 = 2^61`——轻松装进 `usize`，通过所有元素级检查；一乘元素大小、再向上取整到 64 字节对齐，数值就回绕了。溢出只在元素到字节的换算处现形，守卫因此必须钉在那里——所有区域尺寸都要流经的咽喉：

```rust
// gemmkit/src/workspace.rs
fn region_bytes(elems: usize, esize: usize) -> usize {
    elems
        .checked_mul(esize)
        .and_then(|b| b.checked_next_multiple_of(ALIGN))
        .unwrap_or_else(|| workspace_too_large())
}
```

每一步——字节乘积、对齐取整、区域求和、最终的 `A + B` 总量——都带检查，任何溢出都以与驱动器自身尺寸检查相同的“too large”契约 panic。失败即拒绝（fail closed）：荒谬的问题被大声拒绝，而不是败坏内存。驱动器无条件运行它的元素数守卫也是同一个理由，哪怕最终什么都不打包的路线也不豁免——跳过它们就同时跳过了中止，把荒谬的 `k` 送进原地循环里近乎永远地空转。

### 池子、`_with` 与 `no_std`

调用者很少直接看到 `Workspace`，因为线程本地池会透明地提供一个：常规的 `gemm` 调用每线程至多分配一次，之后的每次调用都复用那块缓冲区。这个池是可重入安全的——嵌套的 rayon 可以在一个已经身处 GEMM 的线程上再进入一次 GEMM（一个 worker 在自己的 `for_each` 里阻塞时窃取到另一个 GEMM，或批量并行的 worker 内联执行一个批元素），此时池的 `RefCell` 已被借出，`with_thread_pool` 就在这一次改发一块全新的暂存工作区而不是 panic。打包缓冲区不携带任何结果状态，所以这个回退是不可见的，损失的只是那一次的缓冲区复用。

想要显式控制就用 `*_with` 层：每个入口都有一个变体（`gemm_with`、`gemm_packed_b_with` 等）接受调用者持有的 `Workspace`，从第二次足够大的调用起零堆分配——这是小乘积热循环和延迟敏感代码的工具，`Workspace::with_capacity` 还能免去首次调用的分配尖峰。没有 `std` 时不存在线程本地存储，`with_thread_pool` 就每次调用新建一个工作区；`parallel` 依赖 `std`，所以也没有可重入的线程，想复用的调用者持有一个 `Workspace` 走 `*_with`——这也是 [no_std与WebAssembly](../gemmkit-guide/no_std与WebAssembly.md) 里推荐的用法。
