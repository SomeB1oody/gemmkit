# 扩展点

gemmkit 的变化点是 trait、const 泛型和放在 `OnceLock` 槽里的带类型函数指针——没有宏，没有 `transmute`。这套纪律只为一个目的：让库预期的四种增长（新指令集、新元素类型、新点积指令、新融合变换）都以纯增量代码落地，各自有一份简短、可核对的触点清单，而 driver、打包例程和分块模型原封不动。本页把这四份配方展开成实操走查。它们面向扩展这个 crate 本身的人；不过这些接缝的公开程度足以让最关键的一条——用你自己的内核 family 驱动泛型 driver——在 crate 之外也成立，并且有测试证明这一点。

## 新的 ISA 后端

一个 ISA 后端是一个零大小的 *token* 加上词汇表实现。wasm `simd128` 后端（`gemmkit/src/simd/wasm.rs`）是最近的完整范例，值得从头读到尾——它就是一个文件加几行分发代码。

token 唯一的固有行为是 `Simd::vectorize`，即 `#[target_feature]` 蹦床：运行期 CPU 检测无法与泛型内核上固定的 `#[target_feature]` 搭配，所以每次内核调用都跑在一个带注解的小函数里，`#[inline(always)]` 原语折叠进去，让所有 intrinsic 落进特性已启用的代码生成上下文：

```rust
// gemmkit/src/simd/wasm.rs
impl Simd for Simd128 {
    #[inline(always)]
    unsafe fn vectorize<R>(self, f: impl FnOnce() -> R) -> R {
        #[target_feature(enable = "simd128")]
        fn inner<R>(f: impl FnOnce() -> R) -> R {
            f()
        }
        inner(f)
    }
}
```

清单如下：

1. **token**：在新的 `gemmkit/src/simd/` 模块里定义一个 `Copy + Send + Sync + 'static` 的零大小结构体，按架构 `cfg` 门控，带上面的 `vectorize` 蹦床。
2. **`SimdOps<T>` 实现**，覆盖该 ISA 加速的每个元素类型：寄存器类型、`LANES`，以及原语词汇表（load/store/splat/mul/add/`mul_add`/`fnma`/`reduce_sum`，如果希望融合浮点 epilogue 向量化，还要 `max`/`min`）。词汇表刻意做“厚”，microkernel 才能保持为一个泛型函数；你实现的是原语，从来不是内核。要遵守文档化的契约——simd128 实现用的是 `f32x4_pmax` 而不是 `f32x4_max`，因为 trait 的 `max` 要求 `a` 为 NaN 时返回 `b`（向量与标量 epilogue 在 `ReLU(NaN) = 0` 上的一致）；它把 `mul_add` 写成未融合的 `mul` + `add`，因为 wasm 没有硬件 FMA，而 relaxed-SIMD 替代品在规范上不确定，会破坏可复现性。
3. **tile 几何**：为每个类型选定 `(MR_REG, NR)`，编码为分发模块中各 ISA 包装函数的 const 泛型。这是*唯一*的按 `(类型, ISA)` 旋钮。要明确地做寄存器预算：simd128 的 `f32` 用 2x4（8 个累加器 + 2 个 LHS + 1 个 RHS = 11 个活跃 `v128`），因为 LLVM 的 wasm 后端在约 16 个活跃向量之后开始溢出；NEON 用 4x4，刻意留出约 11 个空闲寄存器。
4. **一个 `Dispatched` 描述符与每条 `select_*` 阶梯一条分支**：记忆化的选择阶梯位于 `gemmkit/src/dispatch/`——`select_f32`/`select_f64`（浮点）、`select_f16`/`select_bf16`（混合）、`select_i8`（整数）、`select_c32`/`select_c64`（复数），外加 map-epilogue 的选择器。每条阶梯分支把普通、预打包、融合入口点和 tile 几何捆成一束，所以加一个 ISA 就是每个受益类型一个描述符常量加一条 match 分支。
5. **一个 `GEMMKIT_REQUIRE_ISA` 名字**：在 `gemmkit/src/dispatch/isa.rs` 里加 `ForcedIsa` 变体及其解析字符串（现有取值：`scalar`、`fma`、`avx512f`、`avx512vnni`、`avx512bf16`、`neon`、`simd128`、`auto`），并遵守“响亮失败”规则——被钉住的 ISA 不受支持时，分发直接 panic 而不是回退，这样一个想测你的内核的 CI 任务不可能悄悄跑在别的内核上通过。
6. **测试基本免费搭车**：`tests/simd_conformance.rs` 直接构造 token，把每个原语对照标量参考检查；再加一个 `env_isa_*` 钉住二进制和一个 CI 任务，分发路线本身就可测了（见[测试与验证](测试与验证.md)）。

你不碰的东西：`driver.rs`、各内核 family、`pack.rs`、`cache.rs`。simd128 后端一个都没改。

## 新的元素类型

元素类型沿两个小 trait 变化（见[标量与内核家族](标量与内核家族.md)）。`Scalar`（`gemmkit/src/scalar.rs`）只声明恒等常量和累加器类型 `Acc`——这是影响最深远的一个决定，它定死了舍入语义（`f16` 选了 `Acc = f32`；`i8` 选了 `Acc = i32`，让整数 GEMM 严格精确）。`KernelFamily`（`gemmkit/src/kernel.rs`）捆起区分一种运算的所有东西：`Lhs`/`Rhs`/`Acc`/`Out` 类型、打包布局和 microkernel。

常常根本不需要新 family。如果新类型是既有累加器之上的窄输入，就在有能力的 token 上实现 `KernelSimd<L, R, A, O>` 拓宽/收窄接缝——拓宽加载、一次收窄存储——像 `MixedGemm<f16>`/`MixedGemm<bf16>` 那样复用泛型 microkernel；同质情形由 blanket 实现覆盖，混合实现与之不可能重叠。真正新的运算形态（平面布局的复数内核、重量化整数 family）才配自己的 `KernelFamily`。

接入公开 API 靠 `gemmkit/src/dispatch/` 下的一个分发模块，每个类型自己的 `OnceLock` 槽：特性检测跑一次，胜出的单态化入口点被缓存，之后每次调用就是一次间接调用。可抄的样板是 `dispatch/mixed.rs`（两个类型、gemv/small-mn/small-k 改道、点积内核选择的细节）或 `dispatch/int.rs`（异质任务类型）。

开闭性质不是口口相传，而是被 `gemmkit/tests/open_closed.rs` 强制执行：它只用公开条目定义了 `NaiveFloat`——一个独立编写、自带打包和朴素标量 microkernel 的 family——用它驱动*未改动*的公开 `driver::run`，并对照 `f64` 参考检查结果。driver 的改动一旦破坏 family 接缝，这个测试连编译都过不了。它也是新 family 的起步模板。

## 点积指令

`vpdpbusd`、`vdpbf16ps` 这类指令把若干深度步折进一条操作，*重塑了累加舍入*——所以它们绝不能以“对可移植 tile 循环的巧妙覆写”的形式到来。接缝相应地被拆开：

- family 声明 `DEPTH_MULTIPLE = Q`（> 1），经 `pack_kgroup_panels`（`gemmkit/src/pack.rs`）打包，把 `Q` 个连续深度步按 lane 连续交织；driver 把 panel 深度向上取整到 `Q`，并保证 k 组不跨切片边界。
- 有能力的 token 覆写 `KernelSimd::dot_accumulate`，从这些 panel 整组整组地消费指令；打包布局是 family 的打包器与覆写 token 之间的私有契约。任何符号修正（VNNI 的 `+128` 技巧及其列和补偿）都在覆写内部完成，返回时累加器持有真实的和。
- `SimdOps::accumulate_tile` 的覆写保留给*调度*层面的改动——保持舍入形状不变：需要显式软件流水线的顺序执行核心，或长度不是编译期常量的可伸缩向量 ISA。它的文档说得很明白：重塑舍入的指令不在这条接缝的范围内，应作为带点积接缝的新 family 到来。`accumulate_tile` 覆写必须保持确定性、与边缘路径舍入一致；默认实现已经能在任何宽乱序核心上打满 FMA 管线，所以留下覆写之前先证明它有收益。

`IntGemmVnni`（对拓宽路径逐位精确，因为整数算术可结合）和 `Bf16DotGemm`（按容差把关，仍在可复现性契约之内）是两个现成例子；[点积内核与深 K 孪生](点积内核与深K孪生.md)有深入讨论。

## 新的融合变换

一个融合变换就是一个 `Epilogue` 实现（`gemmkit/src/kernel/epilogue.rs`）；driver 的 `last_k` 管道、零开销的 `Identity` 默认值、穿过每条特殊路径的路由全都免费（见 [Epilogue 融合](Epilogue融合.md)）。设计工作是选定应用路径并遵守一条硬规则：**向量与标量路径必须逐位一致**——完整 tile 走向量路径，边缘和跨步 tile 走标量路径，同一个输出矩阵混用两者。

- 作用在 `Acc` 类型值上、有自然寄存器形态的变换设 `VECTOR = true` 并实现 `apply_reg`（`FusedEpi` 模式；注意 NaN 和带符号零的语义——`LeakyRelu` 在两种形态里写成完全相同的 `max + slope*min` 组合）。
- 把 `Acc` 收窄成不同 `Out` 的变换设 `VECTOR_STORE = true` 并实现 `apply_store`（`KRequantize` 模式），逐情形论证逐位相等，并用一致性扫描钉住。
- 没有划算向量形态的变换让两个标志都保持 `false`，一切经暂存走标量 `apply`——对任何 tile 形状都正确。但如果标量值可能与快路径的融合存储差一个 ULP，就借用 `MapEpi` 的技巧：设 `VECTOR = true`，把 `apply_reg` 实现为“排空到栈、逐 lane 应用”，让变换看到的永远是普通 `gemm` 本会存储的精确位。

无论走哪条路径，都要在 `gemmkit/tests/epilogue/` 里为新变换补上它自己的 gemm-再-映射等价测试；逐位契约实际上就是在那套测试里被执行的。
