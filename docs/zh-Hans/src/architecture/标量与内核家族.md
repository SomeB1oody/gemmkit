# 标量与内核家族

gemmkit 能相乘的每一种元素类型——`f32`、`f64`、`f16`、`bf16`、`i8`、`Complex<f32>`、`Complex<f64>`，外加只作为重量化输出出现的 `u8`——都流经同一个驱动层、同一套打包框架、同一个缓存模型、同一个并行调度器。全部差异由两个 trait 承载：L0 的 `Scalar`（`gemmkit/src/scalar.rs`）回答"这是什么类型、用什么类型累加"，L1 的 `KernelFamily`（`gemmkit/src/kernel.rs`）回答"这一*类* GEMM 与其他类有何不同"。驱动层对家族保持泛型，从不按元素类型分支；本页讲这道分割线怎么划、为什么划在这里。

## Scalar：只有常量与累加器类型

`Scalar` 刻意做得极小。整个 trait 如下，摘自 `gemmkit/src/scalar.rs`：

```rust
pub trait Scalar: Copy + Send + Sync + PartialEq + 'static {
    /// The type in which products are accumulated. `Self` for `f32`/`f64`
    type Acc: Scalar<Acc = Self::Acc>;
    /// The additive identity
    const ZERO: Self;
    /// The multiplicative identity
    const ONE: Self;
}
```

没有 `Add`，没有 `Mul`，没有转换——只有两个单位元常量和一个关联的累加器类型。省略本身就是设计：所有向量化算术都在 `SimdOps` 里（见 [SIMD 令牌与 ISA 分发](SIMD令牌与ISA分发.md)），epilogue 需要的标量算术则放在几个窄小的侧 trait 上，谁需要谁实现——`f32`/`f64` 实现 `Float`（复数类型借 `num-complex` 的运算符也实现它），`f16`/`bf16` 的加宽/收窄转换在 `NarrowFloat` 上，复数拆分内核的实部/虚部访问在 `ComplexFloat` 上。如果 `Scalar` 本身背着算术，每个新元素类型都欠一整套它未必真正拥有的运算；`i8` 就完全不需要任何算术 trait，它的内核全部通过 SIMD 接缝和精确的 `i32` 整数运算完成。

`Acc` 是混合精度的接缝，表很短：

| 元素类型 | 累加类型 |
|---|---|
| `f32`、`f64` | 自身 |
| `f16`、`bf16` | `f32` |
| `i8`（以及仅作输出的 `u8`） | `i32` |
| `Complex<f32>`、`Complex<f64>` | 自身 |

递归约束 `Acc: Scalar<Acc = Self::Acc>` 让这条链一步封顶（`f16 -> f32 -> f32 -> ...`），泛型代码可以放心引用"累加器的累加器"，不必关心输入有多窄。对同质类型，`Acc = Self` 的分支在编译期塌缩，零开销。

## KernelFamily：一族 GEMM 的全部差异

`KernelFamily` 打包其余的一切：四个元素类型（`Lhs`、`Rhs`、`Acc`、`Out`）、打包布局（`pack_lhs`/`pack_rhs`，写出微面板优先的面板），以及微内核。三个关联常量决定驱动层如何对待一个家族：`OUT_IS_ACC`（部分和能否在深度面板之间经由 `C` 往返——最关键的一个，下文详述）、`FORCE_PACK_LHS`/`FORCE_PACK_RHS`（当打包执行了内核所依赖的变换——复数共轭、点积交织——时置位，驱动层于是永不原地读该操作数），以及 `DEPTH_MULTIPLE`（点积内核的指令组深度；其余家族为 `1`）。

一个家族恰好实现两个微内核方法中的一个。不做融合的家族覆写普通 `microkernel`，继承默认的 `microkernel_epi`——它先断言 `E::IS_IDENTITY` 再转发，是一道"失败即关闭"的护栏：真正的 epilogue 落到不会融合的家族会 panic，而不是被悄悄丢弃。会融合的家族（浮点、混合精度、重量化家族）反过来覆写 `microkernel_epi`，把 epilogue 穿进自己的存储路径；普通方法就此成为死代码，保留 `unreachable!` 默认体。铺块几何刻意*不在*这个 trait 上：`(MR_REG, NR)` 是在分发点按 `(家族, ISA)` 选定的一对 const 泛型，新铺块是一次新实例化，不是新类型。

回报体现在驱动层的签名上：`driver::run::<Fam, S, MR_REG, NR>` 对家族和 ISA 令牌泛型，调用 `Fam::pack_lhs`/`Fam::pack_rhs`/`Fam::microkernel_epi`，全程没有一个针对元素类型的 `if`。新增一类 GEMM 意味着写一个家族，而不是改驱动层。

## 家族清单

目前共有十个家族类型，按代际读——同质、加宽、点积、重量化、复数——接缝一目了然：

| 家族 | 类型（`Lhs`/`Rhs` -> `Acc` -> `Out`） | `OUT_IS_ACC` | `DEPTH_MULTIPLE` | 说明 |
|---|---|---|---|---|
| `FloatGemm<T>` | `T -> T -> T`，`f32`/`f64` | `true` | 1 | 基线：一个泛型微内核覆盖所有 ISA |
| `MixedGemm<N>` | `N -> f32 -> N`，`f16`/`bf16` | `false` | 1 | 经 `KernelSimd` 接缝的加宽 FMA |
| `Bf16DotGemm` | `bf16 -> f32 -> bf16` | `false` | 2 | `vdpbf16ps` 点积内核；两操作数强制打包、k 成对交织 |
| `MixedGemmF32<N>` / `Bf16DotGemmF32` | `N -> f32 -> f32` | `true` | 1 / 2 | f32 输出的深 K 孪生：累加相同，存储为 `f32` |
| `IntGemm` | `i8 -> i32 -> i32` | `true` | 1 | 精确、回绕；加载时符号扩展 |
| `IntGemmVnni` | `i8 -> i32 -> i32` | `true` | 4 | `vpdpbusd` 点积内核，`+128` 符号修正，与 `IntGemm` 逐位一致 |
| `IntGemmQ<O>` / `IntGemmVnniQ<O>` | `i8 -> i32 -> i8` 或 `u8` | `false` | 1 / 4 | 重量化变体（`epilogue` feature） |
| `ComplexGemm<T, CONJ_A, CONJ_B>` | `T -> T -> T`，`c32`/`c64` | `true` | 1 | 拆分（SoA）内核；两操作数强制打包成平面布局；共轭是打包期的符号翻转 |

`FloatGemm` 是参照系：同质类型，一个泛型 `microkernel_impl` 覆盖所有 ISA 和铺块。混合精度与整数家族引入 `Acc != Lhs`，完全倚仗下文的加宽/收窄接缝；点积家族（`Bf16DotGemm`、`IntGemmVnni`）另外换上交织打包布局和硬件点积指令，f32 输出孪生的存在则是为了让深收缩可以重新分块——这些都在[点积内核与深K孪生](点积内核与深K孪生.md)里展开。重量化变体在整数累加之上接一段精确的 `i32 -> i8`/`u8` 重量化，属于 [Epilogue 融合](Epilogue融合.md)的故事。`ComplexGemm` 保持 `Acc = T`，让复数 `alpha`/`beta` 原样穿过驱动层，热循环却经由专用接缝跑在实数分量上——这是[复数拆分内核](复数拆分内核.md)的主题。本页停在清单层面，深入分析在那几页。

## KernelSimd：加宽与收窄的接缝

驱动层对 ISA 令牌的约束是 `S: KernelSimd<Fam::Lhs, Fam::Rhs, Fam::Acc, Fam::Out>`（`gemmkit/src/simd.rs`）。`KernelSimd<L, R, A, O>` 在 `SimdOps<A>`——以 `A` 累加——之上补齐家族在类型边界所需的四个动作：`load_lhs`（加载 `LANES` 个 LHS 值并加宽为一个 `A` 寄存器）、`splat_rhs`（把一个 RHS 标量加宽后广播）、`load_out`（为 `beta != 0` 读取 `C` 时加宽输出值）、`store_out`（把一个 `A` 寄存器收窄为 `LANES` 个输出值，收窄时向最近偶数舍入）。

同质情形零成本：一条覆盖实现 `KernelSimd<A, A, A, A> for S: SimdOps<A>` 把四个方法直接转发到普通的 `loadu`/`splat`/`storeu`，`FloatGemm<f32>` 等家族因此不需要任何 per-ISA 代码。混合家族则按 ISA 添加真正加宽的加载（`f16 -> f32` 走 `vcvtph2ps`，`i8 -> i32` 符号扩展）和真正收窄的 `store_out`。一致性检查是白送的：全等覆盖实现与 `L != A` 的混合实现不可能描述同一组类型。另有两组实现是推导而非逐 ISA 手写的：重量化覆盖实现（`Out = i8`/`u8`，累加侧转发到 `<i8, i8, i32, i32>` 实现）和 f32 输出孪生（`N = f16`/`bf16` 的 `<N, N, f32, f32>`，写成两个具体类型头，因为对 `N` 泛型的覆盖实现无法排除在 `N = f32` 处与同质覆盖实现相撞）。`KernelSimd` 还承载点积接缝（`dot_accumulate`，只有具备点积能力的令牌覆写，默认 `unreachable!`）和向量化重量化存储（`requant_store`，同一模式）。

把这道接缝与驱动层分块绑在一起的常量是 `OUT_IS_ACC`。驱动层通常把 `k` 切成若干 `kc` 面板、让部分和经由 `C` 往返累加（第一个面板之后 `beta = 1`）——当 `Out == Acc` 时这是精确的。当输出比累加器窄时，这种往返会在每个面板边界舍入到 16 位，于是窄类型家族声明 `OUT_IS_ACC = false`，驱动层回应以 `kc = k`：单个深度面板，整条收缩链在 `f32` 寄存器里累加完毕，最后只向窄输出舍入一次。这条单次舍入保证是混合精度结果站得住脚的根基；它的代价——`k` 很大时单面板的 RHS 微面板会撑破 L2——正是 f32 输出孪生要偿还的账，见[点积内核与深K孪生](点积内核与深K孪生.md)。

## 开闭性质的证明

"家族接缝对扩展开放"这一论断没有停留在文字上——它由 `gemmkit/tests/open_closed.rs` 强制执行。这是一个位于 crate 之外、因而只能看见公开 API 的集成测试。测试声明了 `NaiveFloat`：一个刻意朴素的第二浮点家族，与 `FloatGemm` 除公开 trait 外毫无共享——它从零重写微面板打包（crate 内部的 `pack` 辅助函数对它不可见，恰好就是第三方的处境），并提供一个纯标量三重循环 `microkernel`。然后它驱动*未经修改*的泛型驱动层——`driver::run::<NaiveFloat, ScalarTok, 4, 4>`——跑一个 40x33x28 的问题，对照 `f64` 参考实现校验结果。

这个测试的价值主要在于它能编译通过：第二个家族在不改 `driver.rs`、不改 `pack.rs` 的前提下驱动了驱动层，用到的全是公开条目（`gemmkit::kernel::KernelFamily`、`gemmkit::simd::ScalarTok`、`gemmkit::driver::run`、`Workspace`、`Parallelism`）。任何封死接缝的重构——驱动层对某个具体家族分支、依赖某个私有辅助函数、trait 签名里泄漏内部类型——都会先弄断这个文件，然后才伤到下游用户。更完整的测试图景（包括真实家族如何对照 oracle 交叉验证）见[测试与验证](测试与验证.md)；第三方能在这道接缝上搭什么，见[扩展点](扩展点.md)。
