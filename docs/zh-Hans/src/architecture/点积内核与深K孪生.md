# 点积内核与深K孪生

大多数内核家族每次只消费收缩维（k 维）的一个深度步：加载打包 A 面板的一列，广播打包 B 的一个元素，每个累加器寄存器发出一条 FMA（或加宽乘加）。有两个 AVX-512 扩展打破了这个节奏，它们把多个深度步折叠进一条指令：VNNI 的 `vpdpbusd` 把 4 个连续的 `i8` 深度步乘进 16 个 `i32` 通道，AVX-512 BF16 的 `vdpbf16ps` 把 2 个连续的 `bf16` 深度步折叠进每个 `f32` 通道。一条一次消费多个深度步的指令，要求这些深度步在内存中相邻，而且（对浮点来说）会改变累加的舍入方式，所以 gemmkit 给点积内核单独建立了内核家族和 ISA 令牌，而不是藏在共享微内核的分支里。本页依次讲承载它们的两道接缝、两个具体的点积内核，以及两个窄类型家族共享的深收缩路线。

## 为什么点积指令需要独立家族

点积内核与它的加宽（widen）兄弟恰好在两处不同，而这两处分别落在引擎的两条扩展轴上（家族与令牌的划分见[标量与内核家族](标量与内核家族.md)）。打包布局是家族的事：`KernelFamily::pack_lhs`/`pack_rhs` 不接受 ISA 参数，不同的交织方式只能由家族决定——这正是 `Bf16DotGemm` 是 `MixedGemm<bf16>` 的兄弟家族而非其内部分支的原因。内层循环是令牌的事：只有真正具备 `vpdpbusd` 或 `vdpbf16ps` 的 CPU 才能执行它，所以指令藏在一个只有点积能力令牌才覆写的 `KernelSimd` 方法后面。具体的点积家族有：每指令折叠 4 个深度步的 `IntGemmVnni`（及其重量化变体 `IntGemmVnniQ`）、折叠 2 个的 `Bf16DotGemm`，再加上本页末尾讲的两个 f32 输出孪生家族。

## DEPTH_MULTIPLE 与 k 组打包

每指令折叠 `Q` 个深度步的家族声明 `const DEPTH_MULTIPLE: usize = Q`（默认 1）。`gemmkit/src/kernel.rs` 里写明的契约是：家族的打包例程必须写出 `width * kc.next_multiple_of(Q)` 个元素的面板，深度尾部补齐填充，驱动层则按同样的补齐深度为打包面板定步长，两边保持锁步。对所有普通家族来说 `DEPTH_MULTIPLE = 1`，全部 `next_multiple_of` 调用都退化为恒等。

布局本身来自一个共享例程——`gemmkit/src/pack.rs` 里的 `pack_kgroup_panels`，它是交织下标算术的唯一权威。普通的 `pack_panels` 按深度为主序存储面板：每个深度步存 `width` 个连续的先导元素（LHS 是 `mr` 行，RHS 是 `nr` 列）。`pack_kgroup_panels` 则把深度轴按 `Q` 个一组分组，让同一通道的 `Q` 个连续深度值变得连续：面板内第 `g` 组、第 `i` 个通道、组内第 `t` 个位置落在偏移 `g*width*Q + i*Q + t`。这正是一条点积指令要读的形状：一个 64 字节的 A 寄存器覆盖 `LANES` 行乘 `Q` 个连续深度元素，一个 B 组把某一列的 `Q` 个连续深度值当作一次 32 位加载来广播。

这个共享打包器还有两个要点。它接受一个逐元素变换 `xform`——对 bf16 是恒等，对 VNNI 的 A 操作数是 `+128` 偏置——并把所有填充位置（超出块的先导位置和超出 `kc` 的深度位置）填成 `xform(0)`，保证填充与活跃元素一致。另外，由于交织布局无法原地读取，每个点积家族都设置 `FORCE_PACK_LHS`/`FORCE_PACK_RHS`，压过驱动层基于代价的打包决策：点积内核永远要付打包成本，这恰好是下文各个门槛所对冲的开销。`pack.rs` 测试里有一个字节级 oracle，用朴素写法复现该布局，并在宽度尾部、深度尾部、带步长源等各种情形下验证真实例程逐位一致。

在消费侧，`KernelSimd::dot_accumulate` 是各家族用来替代加宽 FMA 循环的接缝。它的默认实现是 `unreachable!`：只有点积能力令牌会覆写它，也只有点积家族会调用它。`Avx512Vnni` 和 `Avx512Bf16` 之所以是独立于 `Avx512F` 的令牌，是因为 `#[target_feature]` 按令牌生效：`_mm512_dpbusd_epi32` 需要 `avx512vnni` 的代码生成上下文，而 `Avx512F::vectorize`（只有 `avx512f`）不提供。该方法接收真实（未补齐）的 `kc`，从补齐深度的面板中读取 `ceil(kc / Q)` 个指令组；一切符号或偏置修正都在内部完成，返回时累加器已持有真实的 `sum_k(A*B)`。折叠之所以放在这个专用接缝而不是通用的 `accumulate_tile` 上，trait 文档写得很清楚：折叠深度步会重塑累加的舍入方式，而 `accumulate_tile` 的契约禁止这样做。

## i8 经 vpdpbusd

`vpdpbusd` 计算的是无符号乘有符号的点积：第一操作数取 u8，第二操作数取 i8。GEMM 要的是有符号乘有符号，修正是代数式的而非逐元素的：LHS 打包把每个字节偏移 `+128` 进入无符号域（`gemmkit/src/kernel/int.rs` 里的 `vnni_a_xform`，使用定义在 `gemmkit/src/simd.rs` 中、只定义一次的常量 `VNNI_A_BIAS = 128`，打包与修正永不失配）。由于 `sum_k((A+128)*B) = sum_k(A*B) + 128*sum_k(B)`，内核通过减去逐列修正项 `128 * sum_k(B[k][j])` 还原真实乘积。`Avx512Vnni::dot_accumulate` 在向量循环之前用一小段标量代码对有符号的打包 B 面板求列和，最后把广播后的修正从每个累加器中减去。填充也是配合好的：A 的填充是 `xform(0) = 128`，其贡献恰被修正项抵消；B 的填充是 `0`，对乘积和列和都没有贡献。

因为 `i32` 累加按环绕（wrapping）进行，而环绕加法在模 `2^32` 下满足结合律，四个一组的重新分组加上偏置修正与升序 `k` 的加宽求和逐位相等：`IntGemmVnni` 与加宽的 `IntGemm` 在任何输入上产生完全相同的输出。ISA 选择因此永远不会改变 `i8` 结果——这比[可复现性契约](设计目标与总体图景.md)要求的还要强。

正是这种可以随时换内核的自由，被小规模并行回退门槛所利用。VNNI 的打包在两个操作数上都是强制的，在小的多线程问题上这道打包壁垒会压过它省下的计算量。所以当选择是自动的、并行度为 `Rayon(n)` 且 `n != 1`、并且 `m*n*k` 低于 `GEMMKIT_I8_VNNI_MIN_PAR_MNK`（默认 `768^3`）时，`dispatch/int.rs` 会把调用交给可原地读取的加宽内核；串行和大规模并行仍然用 VNNI，而强制 `GEMMKIT_REQUIRE_ISA=avx512vnni` 会完全关闭这道门（强制就必须运行指定内核）。预打包 RHS 路径同样绕过此门，原因有二：k 四元组交织的缓冲区只有 VNNI 家族能消费，而且门槛所对冲的打包壁垒在预打包时已一次性摊销——VNNI 的 RHS 打包本来每次调用都是强制的，预打包在这里的收益比任何可原地读取的内核都大。

## bf16 经 vdpbf16ps

`Bf16DotGemm` 是浮点侧的对应者：`DEPTH_MULTIPLE = 2`，两个操作数都按 k 对交织打包（每对是一个 32 位的 `__m512bh` 元素），`dot_accumulate` 每个累加器每个对步发出一条 `vdpbf16ps`。累加之后的一切——alpha 折叠和加宽读 / 收窄写的 epilogue——都通过共享的 `mixed_epilogue` 辅助函数与 `MixedGemm<bf16>` 逐字共用，家族保持 `OUT_IS_ACC = false`，整个收缩在 `f32` 中累加、只向 bf16 舍入一次。

数值故事与 VNNI 有一个本质区别：`vdpbf16ps` 的融合二项点积与两条独立加宽 FMA 的舍入不同，所以点积内核的结果与加宽路径只是容差相等，不是逐位相等。这恰好是引擎一致性标准所允许的：在固定的输入、环境与配置下结果必须可复现，而不是跨内核选择逐位一致。点积内核完全确定，串行、并行、预打包共享同一内核和同一打包布局，因此彼此逐位复现。所以这条路径没有任何规模门槛：只要 CPU 报告 `avx512bf16`，自动选择就优先 `Bf16DotGemm`（选择梯子的注释记录它大约让 bf16 翻倍），也不像 VNNI 那样有小规模并行回退。唯一的例外是三条特殊路径的改道：gemv、`small_mn` 和小 `k` 形状刻意留在 `MixedGemm<bf16>` 的加宽接缝上，因为微小或退化的输出折叠不了什么，点积打包的深度补齐纯属亏损（`i8` 分发出于同样理由把微小形状改道到加宽内核）。

## 深 K 问题

`OUT_IS_ACC = false` 用结构性代价换来单次舍入：驱动层运行 `kc = k`，一个覆盖整个收缩的深度面板，而不是每个同质家族都有的缓存模型 `kc` 切片（见[分块与缓存模型](分块与缓存模型.md)）。一次微块调用要读的 RHS 微面板于是有 `nr * k * sizeof(N)` 字节，一旦它超出 L2，每列条带的 `m/mr` 次微块调用中的每一次都要把它（连同更大的 `mr * k` LHS 微面板）从 L3 或 DRAM 中流过来。这个悬崖可测且陡峭：在 Zen5 9950X（AVX-512F，`nr = 12`）上，`gemmkit/src/cache.rs` 中 `deep_k_engage_bytes` 的注释记录 `f16`/`bf16` 吞吐在 `k = 16384`（384 KiB 微面板）时仍接近峰值，在 `k = 32768`（768 KiB，约为 1 MiB L2 的 0.75 倍）时崩塌。

`gemmkit/src/dispatch/mixed.rs` 里的接入门把这个微面板大小与字节阈值比较：

```rust
let engage_deep_k = NR
    .checked_mul(t.k)
    .and_then(|x| x.checked_mul(core::mem::size_of::<N>()))
    .is_some_and(|bytes| bytes > crate::cache::deep_k_engage_bytes());
if engage_deep_k {
    run_deep_k_twin::<N, Fam::Twin, S, MR_REG, NR>(simd, &t, par, ws);
    return;
}
```

阈值取自 `GEMMKIT_DEEP_KC_BYTES` 旋钮：非零时原样采用；默认的 `0` 则推导为每工作线程有效 L2 容量的一半。选一半而不是整个 L2 是有意的：在被测机器上，整 L2 的门要到 `k ~ 43000` 之后才接入，完全错过悬崖（L2/2 的门在 `k = 32768` 处抓到 2.8 倍、`k = 65536` 处 3.6 倍的 `f16` 加速，同时把 `k = 16384` 及以下留在单面板上——那里孪生路线的差异在噪声之内）。`checked_mul` 链条是失败即关闭的：广播操作数可以带着逻辑上荒谬的 `k` 通过验证，溢出的尺寸必须落回单面板、由其打包尺寸检查拒绝问题，而不是接入一个会对这个 `k` 永远多切片下去的孪生。

## f32 输出孪生

在门槛之上，分发根本不运行窄类型家族。一个小小的 `DeepKTwin` trait 把每个窄家族映射到它的 f32 输出孪生：`MixedGemm<N> -> MixedGemmF32<N>`、`Bf16DotGemm -> Bf16DotGemmF32`——唯一的类型变化是 `Out = f32 = Acc`。这一处变化让 `OUT_IS_ACC` 回到默认的 `true`，驱动层普通的多切片 K 分块原样适用，每个切片的面板重新驻留 L2——这正是全部意义所在。打包布局与累加循环逐字沿用窄家族：`MixedGemmF32` 复用 `pack_panels` 和共享的加宽 FMA 辅助函数，`Bf16DotGemmF32` 复用 `pack_kgroup_panels` 和 `dot_accumulate`；由于累加辅助函数只触及 `KernelSimd` 接缝的输入侧，它们产出的累加器与窄家族将要计算的逐字节相同。

孪生以 `alpha = 1`、`beta = 0` 运行，写入一块 `m x n` 列主序的 `f32` 暂存区，取自一个专用的 `Workspace`（深 K 本来就是大 `k` 场景，一块 `m*n` 的 f32 分配微不足道，独立出来还能让池化的打包工作区留给孪生驱动用）。之后由一次向量化扫描计算 `narrow(alpha*scratch + beta*widen(C))`，逐操作复刻 `mixed_epilogue` 的算术，包括同一个 `store_out` 收窄。

让这条路线不止于近似的，是切片的链接方式。孪生的微内核通过第三个 `KernelSimd<N, N, f32, f32>` 接缝（`gemmkit/src/kernel/mixed.rs` 里的 `twin_seed`）从暂存区为累加器寄存器播种：在累加切片上，它把运行中的部分和加载进寄存器、继续升序 `k` 的累加链，而不是把切片从零求和再事后相加。`f32` 的存取往返是精确的，所以多切片之和就是单面板之和、只是在切片边界处被切开；对 `beta` 属于 `{0, 1}` 的情形，深 K 结果与单面板结果逐字节相同。对一般的 `beta` 则退为容差一致，原因很朴素：单面板在完整微块上用一条 FMA 融合 `beta*C + AB`，在边缘微块上不融合，没有哪一种扫描能同时匹配两者。串行与并行在任何情形下保持逐位一致：孪生驱动的分块与线程数无关，扫描是逐元素的。点积孪生还有一条对齐规则：驱动层把分块 `kc` 向上取整到 `DEPTH_MULTIPLE`，内部切片边界永远不会切开一个 k 对；被切开的对会在收缩中途补零并重组融合点积，有了这条规则，只有最后的短尾被补齐——与单面板情形完全相同。

有三条路线刻意保持单面板：门槛之下的浅 `k`、融合 epilogue 入口、以及预打包 RHS 路径——`DEPTH_MULTIPLE > 1` 的缓冲区要求整个收缩是单个深度切片（驱动层的硬 `assert`，因为违反它会静默地错位读取微面板）。这些等价性声明有直接的测试：`gemmkit/tests/deep_k_narrow.rs` 拨动 `GEMMKIT_DEEP_KC_BYTES`（`1` 在任何 `k` 下强制孪生，`usize::MAX` 强制单面板），在宿主机选中的 ISA 上检验 `beta` 属于 `{0, 1}` 时逐字节相等、一般 `beta` 时容差一致；这个旋钮与其余旋钮一起收录在[调优旋钮](../gemmkit-guide/调优旋钮.md)。
