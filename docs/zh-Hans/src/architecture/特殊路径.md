# 特殊路径

[一次 GEMM 调用的生命周期](一次GEMM调用的生命周期.md)中描述的寄存器分块 driver 建立在一个前提上：每个输出 tile 有足够的工作量来摊销打包、分块和一整个 `MR x NR` 的寄存器累加器。有些形状会彻底打破这个前提：矩阵-向量乘完全没有 tile 复用；`k = 4` 的乘积在打包成本收回之前就算完了；一个 `8 x 8 x 100000` 的收缩会让 driver 把大部分精力花在乘零填充上。L6 层（`gemmkit/src/special/`）正是把这些形状改道到专用内核。每条改道路径都保持三个性质：藏在同样的公开入口之后（`gemm` 及其同族从不暴露走了哪条路）、由[调优旋钮](../gemmkit-guide/调优旋钮.md)阈值门控（校准失准的门可以不重新编译就移动或关闭）、并且维持库的可复现性契约——对下面这些形状，输出在任意 worker 数下逐位一致。

各个门位于每个类型分发入口的顶部，顺序固定。摘自 `gemmkit/src/dispatch/float.rs` 的 `run_typed`（有删节）：

```rust
// gemmkit/src/dispatch/float.rs (run_typed, trimmed)
if (t.n == 1 || t.m == 1) && core::cmp::min(t.m, t.n) <= tuning::gemv_threshold() {
    gemv::run_typed_epi::<T, S, Identity>(/* user frame, before orientation */);
    return;
}
orient_transpose(&mut t);
if small_mn_eligible(&t) || small_mn_pack_eligible(&t) {
    small_mn::run_epi::<T, S, Identity>(/* horizontal dot kernel */);
    return;
}
if t.k <= tuning::small_k_threshold() {
    small_k::run::<FloatGemm<T>, S, MR_REG, NR>(/* one depth panel, in place */);
    return;
}
driver::run::<FloatGemm<T>, S, MR_REG, NR>(/* the general blocked driver */);
```

gemv 在方向归一化之前、也就是用户坐标系里触发；其余的门作用在已归一化的问题上。每条特殊路径都还有对应的融合 epilogue 形态，因此 `gemm_fused` 调用走的路线与其非融合孪生完全相同——这条契约在 [Epilogue 融合](Epilogue融合.md)中展开。

## gemv：受内存带宽支配的边界

`m == 1` 或 `n == 1` 的形状（`gemmkit/src/special/gemv.rs`）每个输出元素只做 `2k` 次浮点运算，却要读一遍 `k` 个矩阵元素：它是内存受限的，全部功夫都在减少 DRAM 流量，而不是调度 FMA。两种取向归约为同一个核心例程：把矩阵（`m == 1` 时先转置）看作 `rows x k` 乘一个 `k` 向量。这个门看形状不看大小：对任何 gemv 形状 `min(m, n)` 都是 1，所以与之比较的 `GEMMKIT_GEMV_THRESHOLD` 旋钮实际上是个开关（设为 0 可把这类形状强制送回通用 driver，那条路同样正确），而不是尺寸上限。

并行度用带宽模型而不是计算 ramp：`Parallelism::resolve_bandwidth` 在一个由 LLC 推导的字节下限以下保持串行（此时单核已经拿到全部缓存带宽），越过下限就直接跳到由核数推导的带宽上限，因为“少数几个 worker”恰好是带宽扩展曲线上最差的一点。输出行按 SIMD 宽度整数倍的粒度切成 panel 分给 worker，每一行完整的 `k` 归约都留在一个 worker 内部，没有拆分归约也没有跨 worker 合并——所以结果在任意 worker 数下逐位一致，比引擎整体的可复现性标准更强。

在一个 worker 的行区间内，布局决定四种策略之一。列主序矩阵走 axpy 形态，它有两个刻意做成逐位一致、只在内存流量上不同的变体：**寄存器分块输出**形态把一个输出行 panel 保持在 SIMD 寄存器中扫完整个 `k`（矩阵和输出各只读一遍）；**普通列外层**形态每列重读一次输出，但矩阵读取是一条完全连续的流。选择依据是输出的缓存驻留性，由 `output_register_block` 判定：当输出（`rows * sizeof`）超出末级缓存的一个比例（普通形态的重读将落到 DRAM）**且** `k` 不超过 `GEMMKIT_K_STREAM_MAX`（默认 32；再大时寄存器分块形态的 `k` 条并发列流会挤爆硬件预取器）时启用寄存器分块。两个变体对每个元素执行同样的升序 `k` 融合累加、同样的按行 SIMD/标量划分，所以调整这个门不会改变任何一位——只改变速度。行主序矩阵走 dot 形态（行按 4 个一组寄存器分块以重叠 FMA 延迟链，每行仍是共享的固定顺序 `dot_contiguous` 归约）；完全跨步的操作数落到标量循环。

混合精度孪生 `run_mixed`（feature `half`）以同样的行划分和同样的逐位一致性论证服务 `f16`/`bf16` gemv，但每次加载都经由 `KernelSimd<N, N, f32, N>` 接缝拓宽到 `f32`，归约在 `f32` 中进行，结果在存储时恰好一次舍入到窄类型。这个单次舍入纪律带来一处不对称：混合 axpy 总是寄存器分块，因为普通列外层形态会按列组重读重写窄输出——变成每组舍入一次而不是每个元素一次。而混合的**融合** gemv 干脆不走这条路：浮点融合 gemv 的融合方式是把已存储的输出重读回来就地映射，这只在浮点输出*就是*累加器时才逐位精确；窄输出在存储时已经舍入，重读再映射等于舍入两次。与其把 epilogue 穿进每个拓宽存储内核，混合融合入口让 gemv 形状留在通用 driver 上，driver 本来就在唯一一次收窄之前于 `f32` 中应用 epilogue（`gemmkit/src/dispatch/mixed.rs`）。

## small-k：单个深度 panel，无可摊销

在很小的 `k`（`gemmkit/src/special/small_k.rs`），整个乘积就是一个深度 panel，driver 的缓存分块模型、工作区划分、尤其是 A/B 打包全都是纯开销：每个打包元素只会被读一次。这条路径以 `kc = k` 直接在 family 的 microkernel 上计算 `C <- alpha*A*B + beta*C`，**就地**读取 A 和 B——不打包、不分块、不碰工作区——同时因为它对 `KernelFamily` 泛型，family 的拓宽/舍入语义免费继承。门是 `k <= GEMMKIT_SMALL_K_THRESHOLD`，一个按架构拆分的机器校准值（x86 默认 16，aarch64 默认 8：NEON 的窄 tile 打包足够便宜，driver 更早胜出）。

就地读取有前置条件，不满足时路径退回 driver（仍然正确，只是调度不同）：microkernel 要求 LHS 行单位步长，所以 A 必须列主序（`rsa == 1`）；`FORCE_PACK_*` family（复数，其打包会把数据*变换*成平面布局）从构造上就无法就地读取；`k` 超过硬上限 `SMALL_K_MAX = 32` 会溢出该路径唯一使用的栈缓冲——为底部不完整行 tile 准备的零填充 panel（microkernel 总是加载满 `mr` 行，所以那个 tile 必须打包）。工作按输出 tile 划分，每个 tile 由一个 worker 完成完整的一遍 `k`，worker 数用带宽模型封顶（小 `k` 下 `m*n` 的输出写入占主导）：与 gemv 一样，任意 worker 数下逐位一致。

## small-mn：小输出的水平点积

当 `m` 和 `n` 都远小于 microtile 而 `k` 很长时（`gemmkit/src/special/small_mn.rs`），driver 会把微小的行列 tile 填充成完整的 `MR x NR` microtile，算的大多是填充。这条路径改为把每个输出元素算成一次水平 SIMD 点积 `C[i,j] = alpha*<A[i,:], B[:,j]> + beta*C[i,j]`，沿收缩方向流式读取。输出按 `4 x 4` 的累加器 tile 做寄存器分块：16 条独立 FMA 链在整个 `k` 扫描中保持在飞行状态，每个 A 行和 B 列每 tile 只加载一次——与 gemv dot 形态相同的延迟隐藏技巧，推广到一个小网格。维度门是 `m, n <= GEMMKIT_SMALL_MN_DIM`（默认 16；aarch64 为 32——零拷贝档在 32 及以下都占优）且 `k` 高于 small-k 阈值，两条小形状路径就此瓜分 `k` 轴。

内核要求两个操作数都沿 `k` 单位步长：A 的行连续（`csa == 1`，行主序 A）、B 的列连续（`rsb == 1`，列主序 B）。这是零拷贝档。最常见的两种布局恰好各缺一边——全行主序缺 B，全列主序缺 A——于是第二个与之互斥的门（`k > GEMMKIT_SMALL_MN_PACK_MIN_K`，默认 16）启用预打包档：`prepack_operands` **只把不合格的那个操作数**拷贝进 `k` 连续的工作区暂存，然后同一个内核以单位步长在其上运行。拷贝触及 `m*k`（或 `n*k`）个元素，对比 `m*n*k` 的点积工作量，是约 `1/n`（或 `1/m`）的税，水平内核的收益远超于此，所以跨步布局的小 `m,n` 形状仍然胜过退回 driver 的填充 microtile。暂存把每条线的步长向上取整到*奇数*个缓存行（`packed_line_stride`）：自然步长 `k` 在 `k` 为 2 的幂时会让所有打包行落进同一个 L1 组，重读崩塌（修复前实测在 `k >= 1024` 时比 driver 慢约 3 倍）。预打包是纯粹的重排——同样的值、同样的每行顺序——因此打包路线与合格布局路线逐位一致。混合精度（`f16`/`bf16` 拓宽到 `f32`、每格恰好一次舍入）与整数（`i8 -> i32`，回绕算术因此与 driver 逐位一致）两个同胞共享 tiling、预打包助手和可复现性论证。

## batched：编排层，不是内核

批量 GEMM（`gemmkit/src/special/batched.rs`）刻意不做新内核：每个批元素都重新经过完整的单 GEMM 引擎分发，所以批量调用自动与 driver、gemv、small-k、small-mn 组合。这一层添加的是调度，由 `Parallelism::resolve_batch` 每次调用决定一次：

- **`BatchParallel`**——元素足够填满 worker：每个 worker 串行、缓存热地跑完整的 GEMM，整个批只付一次 fork/join 而不是每个元素一次。这正是“大量小矩阵”这一动机工作负载的正确模型。
- **`SequentialInternal`**——元素少而大、DRAM 受限：顺序循环整个批，轮流给每个元素完整的引擎并行度。x86 上当元素溢出单核 L2 时启用；aarch64（共享簇 L2、高统一带宽）上当每批 worker 的份额 `elem_bytes / batch` 超过 `GEMMKIT_SEQ_INTERNAL_BYTES_PER_WORKER`（默认 128 KiB）时启用。这个方案会把一个元素拆到多个 worker 上，因此只对 `m, n > 1` 的形状开放——driver / small-k / small-mn 路线在单个 worker 内完成每个输出的归约，在与线程数无关的分块下串行与并行逐位一致；gemv 只承诺可复现性，故被排除。
- **Serial**——总工作量低于门限，或没有可用线程。

由于元素相互独立、串行与批级并行方案从不拆分任何元素，这两种方案下的批量结果在任意 worker 数下逐位一致，与每个元素内部走哪条路线无关。带步长的批量入口（`gemm_batched`，以及把同一个共享 epilogue 穿过同一骨架的 `gemm_batched_fused`）共用一份调度实现，普通与融合形态不会漂移。对元素形状各异的批，指针数组形式 `gemm_batched_ptr_unchecked` 接受一个 `GemmProblem` 描述符切片——每个元素有自己的维度、步长和指针——采用更简单的 `resolve_batch_flat` 策略（整个 GEMM 分给 worker，绝不做元素内拆分，因为没有统一的驻留性可测）；`gemm_batched_slice` 是它经过校验的安全孪生。这一切面向用户的视角见[批量 GEMM](../gemmkit-guide/批量GEMM.md)与[小形状与 GEMV](../gemmkit-guide/小形状与GEMV.md)；这些路径依赖的 worker 数机制见[并行执行](并行执行.md)。
