# 设计目标与总体图景

gemmkit 是一个纯 Rust 的 GEMM 引擎：在带显式步长的 `&[T]` 切片视图（或带 `isize` 步长的裸指针）上计算 `C <- alpha*A*B + beta*C`，并在运行时选择当前机器上可用的最优指令集——x86-64 上的 AVX-512 或 FMA/AVX2（外加专门的 VNNI 与 BF16 点积内核）、aarch64 上的 NEON、wasm32 上的 `simd128`，以及在任何平台都兜底的可移植标量路径。整个工作空间基于 edition 2024，`rust-version` 为 1.89，许可证为 MIT OR Apache-2.0。

API 表面由三类调用者共同塑造。应用代码走安全的切片入口（`gemm`、`gemm_fused`、`gemm_i8` 等），在任何 unsafe 代码运行之前完成全部校验。线性代数库——随仓库发布的 `ndarray`、`nalgebra`、`faer` 适配器，以及任何按同样方式构建的库——走 `*_unchecked` 层，它信任调用方自己的不变量，并接受安全层无法表达的内存布局。而受限的部署环境则得到一个可以 `#![no_std]` 构建、零强制依赖的核心，一路支持到编译期开启 SIMD 的 wasm32。本章其余内容都可以从四条设计信条推导出来；它们在 [ARCHITECTURE.md](https://github.com/SomeB1oody/gemmkit/blob/master/ARCHITECTURE.md) 的 "Goals and constraints" 一节里有紧凑的表述，这里展开每一条背后的理由。

## 边界上的安全性

各个受检入口在触碰任何 unsafe 代码之前都会先运行 `validate_gemm_views`（`gemmkit/src/api.rs`），其 panic 清单刻意做到详尽：

- **形状不匹配**：`A.cols != B.rows`、`A.rows != C.rows` 或 `B.cols != C.cols`，每条消息都写明两个不一致的数字。
- **视图越界**：对 A、B、C 各自计算步长所能到达的最高偏移（`extent`），并与切片长度比对；一个需要的元素数超过切片实际长度的视图会带着确切的差额 panic。
- **负步长**：安全层直接拒绝，消息指向 `gemm_unchecked`。带负步长的 `&[T]` 视图必然要寻址到切片起点之前，安全层基于切片的范围推导根本无法为它担保。
- **自混叠的输出**：C 的步长若把两个不同的 `(i, j)` 映射到同一偏移——零步长是最常见的情形——在 A、B 上没有问题（广播输入只被读取），但在 C 上会 panic：并行驱动假设输出 tile 互不相交，通过这样的视图写出会构成一个从纯安全代码就能触达的数据竞争。
- **C 与 A/B 重叠**：按字节区间比较，因此即使在异构的整数入口（C 是 `i32`，A/B 是 `i8`，元素大小不同）也保持精确。融合入口还会额外校验偏置长度（`PerRow` 为 m、`PerCol` 为 n）以及偏置与 C 不相交。
- **问题大到无法计量**：广播步长允许逻辑维度逼近 `isize::MAX`，内部打包缓冲区的尺寸计算可能让 `usize` 溢出；所有这类乘积都在元素数到字节数的必经关口（`Workspace::regions`）以“失败即封闭”的方式 panic，而不是回绕后分配不足。

panic 的措辞本身也是被测试的契约：正确性测试断言这些字符串的精确内容，所以修改错误消息是一个刻意且可见的动作。

`*_unchecked` 层之所以存在，是因为这套校验只在一个边界上才有意义。适配器直接从 `ndarray`/`nalgebra`/`faer` 类型中取出指针和步长，这些类型自身的不变量已经保证了有效性——重复检查是纯粹的开销，而且基于切片的检查根本无法表达适配器需要的东西：一个反转的 `ndarray` 视图有负步长，基指针指向分配区的中间，这两者对裸引擎都是合法且健全的。于是安全成本恰好只付一次：要么由 gemmkit 的校验器付，要么由调用方的类型系统付，绝不重复。unchecked 入口是带有明确文档契约的普通 `unsafe fn`；使用指南在 [Unchecked 层](../gemmkit-guide/Unchecked层.md)中有专门介绍。

## 可复现、而非逐位相同的并行结果

承诺是：在固定的输入、环境和配置下，输出与工作线程数无关，完全一致。支撑它的机制有三个。分块只由缓存模型计算，与线程数无关，所以每次运行对问题的切分都相同。每个输出元素都由同一个工作线程沿完整深度从头归约到尾——引擎中任何地方都没有拆分的归约。打包出的字节不取决于是谁打的包，因此按需调度可以把任何 tile 交给任何线程。哪个线程算哪个 tile 每次运行都会变，结果永远不变。

同样重要的是**没有**承诺什么。串行与并行逐位相同不是契约——它在今天的驱动路径上恰好成立，因为串行和并行跑的是同一套内核、同一套分块，但没有任何东西把它钉死。跨配置的逐位一致也明确不在承诺内：改一个调优旋钮，分块随之改变，浮点求和顺序就可能合法地改变。同一类型不同内核之间的逐位一致同样不在承诺内：bf16 的 `vdpbf16ps` 点积内核相对拓宽 FMA 路径重塑了累加的舍入方式，用容差而非相等来验收。

为什么把线画在这里？永久承诺串行/并行逐位相同会禁止真正有价值的工程——把相邻深度对融合进一条指令的点积内核，或将来考虑并行度的分块选择——却换不来任何用户能跨机器、跨版本依赖的东西。固定配置下的可复现性才是测试能断言、部署能依赖、引擎又能在演进中守住的性质。在能低成本给出更强保证的路径上，就给出更强保证：gemv 按输出行划分，跨任意线程数逐位一致；`i8` 整数路径是精确算术，其 VNNI 点积内核与拓宽内核逐位一致。

## 变化点上无宏、无 `transmute`

引擎沿三条轴变化——指令集、元素类型、运算家族——每条轴都是一个普通 trait：ISA 对应 `Simd`/`SimdOps`，元素类型对应 `Scalar`，家族对应 `KernelFamily`。分发槽是缓存在 `OnceLock` 里的带类型函数指针，微 tile 几何是在分发点选定的一对 const 泛型。一个“内核变体”实际长什么样，看 `gemmkit/src/dispatch/float.rs`：

```rust
unsafe fn gemm_f32_fma(t: Task<f32>, par: Parallelism, ws: &mut Workspace) {
    // MR = 2*8 = 16, NR = 6 -> 12 acc + 2 lhs + 1 rhs = 15 YMM
    unsafe { run_typed::<f32, Fma, 2, 6>(Fma, t, par, ws) }
}

unsafe fn gemm_f32_avx512(t: Task<f32>, par: Parallelism, ws: &mut Workspace) {
    // MR = 2*16 = 32, NR = 12 -> 24 acc + 2 lhs + 1 rhs = 27 ZMM
    unsafe { run_typed::<f32, Avx512, 2, 12>(Avx512, t, par, ws) }
}
```

这就是每个 (类型, ISA) 组合的全部专属代码：一个点名令牌和 tile 的包装函数。另一条路——宏展开或手抄的每 ISA 内核，C 语言 BLAS 的传统——因为可审阅性和扩展成本被否决了。用 trait 加 const 泛型，全库只有一个泛型微内核可读、可单步调试、可修复；一次调度改进落地一处，所有 ISA 一起受益。编译器对每个单态化实例做完整类型检查，而且 `OnceLock` 槽里放的是**带类型**的函数指针而非类型擦除的指针，签名漂移是编译错误，不是潜伏的 `transmute` 缺陷。扩展遵循同样的形状：新 ISA 是一个零大小令牌、它的 `SimdOps` 实现、每个选择阶梯里的一条分支；新元素类型是一个 `Scalar` 实现、一个家族（或通过拓宽/收窄接缝复用现有家族）加一个分发槽。驱动、打包、分块永远不动——这条性质由一个测试（`gemmkit/tests/open_closed.rs`）强制执行：它声明第二个平凡家族去驱动驱动器。后续两页 [SIMD 令牌与 ISA 分发](SIMD令牌与ISA分发.md)和[标量与内核家族](标量与内核家族.md)会逐条接缝展开。

## `no_std` 与零强制依赖的核心

关掉默认 feature 后，核心 crate 是 `#![no_std]`，只需要 `core` + `alloc`，不依赖任何 crate。每个可选 feature 至多引入一个依赖：

| Feature | 引入的依赖 | 换来的能力 |
|---|---|---|
| `std`（默认） | `raw-cpuid`（仅 x86/x86-64 目标） | 运行时缓存与 CPU 特性探测、`GEMMKIT_*` 环境变量旋钮、线程本地工作区池 |
| `parallel`（默认） | `rayon` | `Parallelism::Rayon` 多线程 |
| `half` | `half` | `f16`/`bf16` 混合精度 GEMM |
| `complex` | `num-complex` | `c32`/`c64` 复数 GEMM |
| `int8` | 无 | `i8 -> i32` 整数 GEMM |
| `epilogue` | 无 | 融合偏置/激活/映射 epilogue（重量化还需同时开启 `int8`） |
| `wasm_threads` | 除 `parallel` 外无 | 线程化 wasm 上显式定容的 rayon 池 |

没有 `std` 时，编译期 target feature 取代运行时 CPU 探测，环境变量旋钮关闭（程序化的 `tuning::set_*` 仍然可用——它们是纯原子操作），线程本地池换成按调用传入的工作区。要点在于：位于栈这么底层的一个内核，不应该把依赖策略强加给宿主——嵌入式或 wasm 部署得到的是同一个驱动、同一批家族、同一份可复现性契约，少掉的只有真正需要操作系统的那部分机制。实操指南见 [no_std 与 WebAssembly](../gemmkit-guide/no_std与WebAssembly.md)。

## 工作空间地图

五个 crate 以 0.1.0 版本同步发布，另有一个刻意独立成根的模糊测试 crate：

| 路径 | Crate | 角色 |
|---|---|---|
| `gemmkit/` | gemmkit | 核心 GEMM 引擎——本章描述的全部内容 |
| `gemmkit-ndarray/` | gemmkit-ndarray | 面向 `ndarray`（>= 0.17.1）视图的零拷贝适配器 |
| `gemmkit-nalgebra/` | gemmkit-nalgebra | 面向 `nalgebra` 0.35 矩阵的零拷贝适配器 |
| `gemmkit-faer/` | gemmkit-faer | 面向 `faer` 0.24 矩阵的零拷贝适配器 |
| `gemmkit-tune/` | gemmkit-tune | 安装期自动调优程序，输出一份 `GEMMKIT_*` 环境变量配置 |
| `gemmkit/fuzz/` | gemmkit-fuzz | cargo-fuzz 目标；仅限 nightly，被排除在稳定工作空间之外 |

适配器有意做薄：各自直接从宿主库的原生视图里取出矩阵指针和步长——C 序、F 序、一般步长、反转步长，全程零拷贝——然后转发给 `*_unchecked` 引擎，用宿主类型的不变量替代安全层本要做的校验。每个适配器把同名 Cargo feature（`parallel`、`wasm_threads`、`half`、`complex`、`int8`、`epilogue`）转发给 gemmkit，因此 feature 故事处处一致。各适配器章节（[ndarray](../gemmkit-ndarray/在ndarray中使用gemmkit.md)、[nalgebra](../gemmkit-nalgebra/在nalgebra中使用gemmkit.md)、[faer](../gemmkit-faer/在faer中使用gemmkit.md)）覆盖了它们的完整表面。

`gemmkit-tune` 是进程外的校准器。引擎里每个启发式阈值都是运行时旋钮（见[调优旋钮](../gemmkit-guide/调优旋钮.md)），而编译进去的默认值是在一台机器上校准的，调优器的意义就是在你的机器上重做这次校准：在部署主机上运行一次这个二进制，它对每个旋钮在一组探测形状上扫描，写出一份由 `export GEMMKIT_*=...` 行组成的 `gemmkit-tune.env` 配置，在启动应用前 source 即可。不需要重新编译，也没有构建期耦合——调优器与库之间唯一的契约就是文档化的环境变量表面，由 `tuning::knob_env_names` 注册表保证：调优器的扫描表会与它做断言比对。实操指南见 [gemmkit-tune 章节](../gemmkit-tune/使用gemmkit-tune调优.md)。

模糊测试 crate 被有意放在工作空间之外：cargo-fuzz 需要 nightly（build-std、AddressSanitizer），排除它可以让 `cargo test --workspace` 和 MSRV 构建始终留在 stable 上。

## 本章与 ARCHITECTURE.md 的关系

仓库里的 `ARCHITECTURE.md` 是紧凑的地图：分层表、调用路径、各条接缝、每个子系统一节，为另一个窗口开着代码的读者而写。本书这一章是同一素材的导览版——相同的层号，相同的文件引用，但留出了讲清理由、被否决的备选方案和完整示例的篇幅。两者若有出入，以代码为准，且两份文档都算有 bug。接下来请读[分层结构](分层结构.md)看静态骨架，再读[一次 GEMM 调用的生命周期](一次GEMM调用的生命周期.md)看动态过程。
