# 小形状与GEMV

gemmkit 核心的寄存器分块 driver，是为每个输出 tile 有足够工作量来摊销打包、缓存分块和一整个 `MR x NR` 累加器的乘积而设计的。有些形状会彻底打破这个前提：矩阵-向量乘完全没有 tile 复用，`k = 4` 的收缩在打包收回成本之前就算完了，而一个 `8 x 8 x 100000` 的乘积会让 driver 把大部分精力花在乘填充上。对这些形状，driver 是错的工具，引擎会悄悄绕开它。

没有什么要去开启。同一个 `gemm`（以及 `gemm_i8`、`gemm_fused`、`gemm_map`）入口会在分发的最顶端检查形状与步长，遇到合适的形状就改道到一个特殊内核，否则落回通用 driver。每一次改道都藏在同样的公开入口之后，保持库的可复现性契约（输出在任意 worker 数下逐位一致），并由一个你无需重新编译就能移动或关闭的[调优旋钮](调优旋钮.md)门控。你从不直接调用这些路径；你只是因为把形状写得自然、而不去手搓点积循环，就享受到它们的好处。

## gemv：受内存带宽支配的边界

`m == 1` 或 `n == 1` 的形状是一个矩阵-向量乘。每个输出元素只做 `2k` 次浮点运算，却要读 `k` 个矩阵元素，所以算术是琐碎的，全部功夫都在减少 DRAM 流量。专用的 gemv 路径（`m == 1` 与 `n == 1` 归约为同一个核心例程，把矩阵——`m == 1` 时先转置——看作 `rows x k` 乘一个 `k` 向量）对每种布局都正确，并对连续布局做了向量化。

有两条性质对调用者要紧。其一，gemv **对 worker 数逐位一致**：每个输出元素由一个 worker 在一趟 `k` 扫描里算出，而输出行的划分从不切分一个元素的归约，所以加线程绝不改变任何一位。其二，gemv 有**自己的一套并行策略**。因为它受带宽支配，worker 数由一个带宽模型来限制，而非通用 driver 用的那条计算爬坡：过了那几个能饱和 DRAM 的核心之后，更多 worker 就不再有用，只会徒增 fork/join 和共享缓存争用。有两个旋钮暴露这一点：`gemv_parallel_bytes`（一个字节下限，低于它 gemv 因数据缓存驻留而保持单线程）和 `gemv_thread_cap`（gemv worker 数的硬上限），此外还有 `gemv_threshold`——由于 gemv 形状总满足 `min(m, n) == 1`，它实际上是个开关。

## 小 k 路径

当收缩 `k` 较小时（在 `small_k_threshold` 及以下，x86 默认 16、aarch64 默认 8），整个乘积就是单个深度面板，而 driver 的打包无从摊销，因为每个打包元素只被读恰好一次。小 `k` 路径（瘦长与低深度形状：gevv、rank-`k` 更新、高瘦乘积）直接在家族的微内核上计算 `C`，**就地**、不打包地读 `A` 和 `B`，一趟算完。它免费继承了家族的加宽 / 偏置 / 共轭 / 舍入行为，并对任意 worker 数都与串行运行逐位一致。它需要列主序的 `A`（行单位步长，`rsa == 1`）；当这不成立时，在极小 `k` 下打包很少能摊销，所以该路径退回通用 driver，仍然正确。

## 小 m,n 路径

镜像的情形是一个很小的输出配上一个很长的收缩：`m` 与 `n` 都远低于微 tile（在 `small_mn_dim` 及以下，默认 16），而 `k` 很长。driver 会把很小的行 tile 和列 tile 填充到一整个微 tile，算的大多是填充。这条路径改为把每个输出算成一个水平点积，`C[i,j] = alpha * <A[i,:], B[:,j]> + beta * C[i,j]`，沿收缩流式跑 SIMD，没有任何分块或取向机制，并对这个小输出网格做寄存器分块，让好几条独立的 FMA 链同时在飞。

这个水平内核需要两个操作数都沿 `k` 单位步长：`A` 的行连续（`csa == 1`，即**行主序 A**）以及 `B` 的列连续（`rsb == 1`，即**列主序 B**）。两者都成立时，该路径就地零拷贝地读 `A` 和 `B`，即快路径。两种最常见的布局各自恰好缺一边（全行主序缺 `rsb`，全列主序缺 `csa`），于是一次内部预打包把**仅那个不达标的操作数**拷进 `k` 连续的暂存区一次，再在其上跑同一个水平点积。这次拷贝是 `m*k`（或 `n*k`）次读，相对 `m*n*k` 的乘积而言，是大约 `1/n`（或 `1/m`）的税，被水平的收益远远盖过，所以一个带步长的小 `m,n` 形状仍胜过落回 driver 的填充微 tile。这个预打包档在 `k` 越过它自己的旋钮 `small_mn_pack_min_k`（默认 16）时启用，与零拷贝档的 `small_k_threshold` 分开。这条路径同样对 worker 数逐位一致，把每个输出算成在一块不相交 tile 上的一次定序归约。

## 实用建议

优先用现成的入口，而非手写点积循环。如果你的问题是一个矩阵-向量乘、一次 rank-`k` 更新，或长收缩之上的一格小输出，`gemm` 早已带着为之调好的内核，连同带宽感知的线程策略和位级可复现性——这些都是手写循环得从头再造的。布局是你手里唯一的杠杆：对水平的小 `m,n` 路径，**行主序 A 和列主序 B** 沿 `k` 单位步长、命中零拷贝快路径；对小 `k` 路径，**列主序 A** 留在就地路径上。其它任何布局仍然可用，只是要么付小 `m,n` 的预打包拷贝，要么落回通用 driver。

这里提到的每个阈值都是可调旋钮，按每次调用从参数、程序化 setter、`GEMMKIT_*` 环境变量，或校准过的编译期默认值中解析。如果某次改道对你的机器失准，或你想把某个形状强制送上通用 driver，[调优旋钮](调优旋钮.md)一章完整讲解了 `gemv_threshold`、`small_k_threshold`、`small_mn_dim`、`small_mn_pack_min_k`、`gemv_parallel_bytes`、`gemv_thread_cap` 与 `k_stream_max`。特殊路径的*内部*（每个内核为何长成那样）在架构一章的[特殊路径](../architecture/特殊路径.md)里。
