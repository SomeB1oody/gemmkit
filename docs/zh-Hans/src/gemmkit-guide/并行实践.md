# 并行实践

每个 GEMM 入口都以 `Parallelism` 作为最后一个参数。它是一个只有三种实用模式的小枚举，而用得好与用得糟之间的区别，多半在于理解 `auto` 模式替你做了什么决定、以及何时该由你自己接管方向盘。

## 三种模式

```rust
pub enum Parallelism {
    Serial,       // 单线程
    Rayon(usize), // 至多 n 个线程的 rayon；Rayon(0) 自动探测
}
```

`Serial` 把整次调用跑在调用线程上。`Rayon(n)` 请求至多 `n` 个工作线程。`Rayon(0)` 是自动模式，也是 `Default`，因此 `Parallelism::default()` 给你自动模式。`Rayon(n)` 里的 `n` 是分区数的上限，而非用满它们的承诺——工作量不足 `n` 个块、或核心数少于 `n` 的问题，会分得更少。

## 自动模式究竟做了什么

自动不是“用满所有核心”，而是叠加在问题规模上的两个决定。

首先是一道**工作量闸门**。当 `m*n*k` 总工作量低于某阈值（`GEMMKIT_PARALLEL_THRESHOLD` 旋钮，默认 `48*48*256`）时，无论如何都保持串行——在小矩阵上，fork/join 开销会淹没任何收益。这道闸门先于其余一切执行，因此对显式的 `Rayon(n)` 同样生效：低于它时，`Rayon(8)` 仍然跑在单线程上。

越过闸门后，自动模式会**让工作线程数随总工作量扩展**，而非直接跳到满核心数。具体来说，它以 `m*n*k` 除以 `GEMMKIT_PAR_MNK_PER_WORKER`（默认 `2_000_000`——即每这么多工作量配一个工作线程）为目标，再用机器的核心数和可用作业块数封顶，下限为一。之所以按工作量而非维度计，是因为实测最优值跟随的是总 flops 而不是线性尺寸：在一颗 Ryzen 9950X 上，`128^3` 串行跑得最快，而 `384^3` 已经想要用满每一个硬件线程——这样的跨度没有任何单一的线性维度步长能同时拟合两端。于是小乘积只用少数几个工作线程，大乘积用很多个；把 `GEMMKIT_PAR_MNK_PER_WORKER` 设为 `0`（其行为等同于 `1`）则会让任何越过串行闸门的问题强制跑满全宽。

## 显式数量

`n > 0` 的 `Rayon(n)` 绕过爬升启发式，请求恰好 `n` 个分区——但出于安全，仍会被机器的核心数（`available_parallelism`）和问题实际切分出的作业块数封顶。因此在一台 16 核机器上计算小乘积时，`Rayon(1000)` 不会过度订阅；它会坍缩到机器与工作量所能吸收的程度。正是这种精确性，让测试套件与扩展诊断使用显式数量：`Rayon(4)` 给你四路分区（当有那么多工作量、那么多核心时），而非一个启发式猜测。当你已经测量过自己的工作负载、知道甜点区在哪里，或者当你想要跨运行可复现的分区以做基准测试时，就用显式数量。

## gemmkit 如何使用 rayon 池

在原生目标上，gemmkit 不构建也不拥有 rayon 线程池。当它决定运行 `k` 个分区时，会经由 rayon 的并行迭代器把它们驱动在*环境*池上——默认即 rayon 的全局池。这带来一个有用的后果：如果你把一次调用包在自己池的 `install` 里，这次 GEMM 的工作线程就会跑在你的池上。

```rust
let pool = rayon::ThreadPoolBuilder::new().num_threads(4).build().unwrap();
pool.install(|| {
    gemm(1.0, a, b, 0.0, c, Parallelism::Rayon(0)); // 跑在 `pool` 上
});
```

gemmkit 选择的工作线程*数量*仍受 `available_parallelism`（整台机器）约束，而这些分区会由 rayon 的工作窃取调度器分布到当前池所拥有的那些线程上——因此一个更小的自定义池只是把同样的分区跑在更少的线程上。一次调用内部的工作分配是按需驱动的：工作线程从一个共享的无锁游标上领取连续的块，因此异构部件（P/E 核心布局）上更快的核心会按比例多领一些，而不是所有人都等最慢的那个。线程化 wasm 的情况有所不同——那里 gemmkit *确实*会给一个专用池定尺寸——见[no_std 与 WebAssembly](no_std与WebAssembly.md)。

在这套环境池的故事之下，还有第二套机制，仅限原生（x86_64 与 aarch64 上默认开启）：gemmkit 还保留最多 `GEMMKIT_POOL_CLASSES`（x86_64 默认 2，aarch64 默认 1）个私有的、持久化的池，尺寸恰好卡在机器宽度的减半档位上——在一台 32 线程的机器上是 16 与 8 线程；在 14 核的 M4 Max 上是单个 7 宽档位——首次用到时才惰性构建，此后不再重建。自动模式会把它的 worker 数精确卡在其中某个档位上，而不是 fork 进全宽的全局池，因为一次 fork 的开销跟的是池的空闲余量（池所拥有的线程数超出实际参与工作的那部分），一个小规模 GEMM 一旦落进全宽池，就会被这份空闲余量拖累。这一切都不会改变你已经知道的东西：只要调用跑在你自己的池里，它就绝不会触发——`install` 过的调用依然被完全尊重，绝不会被改道到某个档位池——显式的 `Rayon(n)` 仍然恰好拿到 `n` 个 worker，只是会被路由进能容纳它的最小档位池。真正改变的是闲置内存：默认情况下，一个 x86_64 进程现在会额外常驻大约 24 个线程（16 宽与 8 宽这两个档位池），一个 aarch64 M4 Max 进程则常驻 7 个（它唯一的半宽档位），与全局池一起，在小规模 GEMM 需要它们之前都处于休眠。把 `GEMMKIT_POOL_CLASSES` 设为 `0` 即可彻底关闭档位池，让每次调用都回落到环境池。

## 精确表述可复现承诺

对**固定的输入、环境与配置，输出与工作线程数无关地一致。**这就是契约。它之所以成立，是因为 `kc`、`nc` 与固定的深度面板顺序——唯一塑造每个输出元素求和方式的东西——的计算独立于将由多少线程来运行它们，而每个输出元素都由单个工作线程沿完整收缩深度从头到尾归约，因此没有归约拆分、其顺序会依赖调度。（扁平作业列表本身在不同宽度下*并非*严格一致——宽工作线程数会缩小 `mc` 好让列表保持足够深——但 `mc` 始终是 `mr` 的倍数，所以微 tile 集合及其数值都不变。）打包出的字节不取决于由谁来打包。哪个工作线程计算某个 tile 会因运行而异；数值结果不会。

**不**被承诺的是 `Serial` 与 `Rayon(n)` 之间的逐位相同。它今天在驱动路径上恰好成立——串行与并行运行同一个内核——但你应当据以构建的保证是固定配置下的可复现性，而非串行对并行的逐位相等。若你需要跨机器或跨配置的逐位相等，这里给不了；浮点 GEMM 对顺序敏感，而配置（ISA、分块、线程上限）是固定输入的一部分。整数 `gemm_i8` 是例外：它跨 ISA、跨工作线程数逐位相同，因为 `i32` 加法与顺序无关。

## 何时该选 Serial

在三种情形下伸手去拿 `Serial`。其一，**小问题**：低于工作量闸门时自动模式本就是串行，但显式传入 `Serial` 还会彻底跳过 `available_parallelism` 探测与 fork 机制，这在一串微小 GEMM 的紧循环里有可测量的更低成本。其二，**当外层并行由你掌控时**：如果你已经在一个 rayon 池上运行许多独立的 GEMM、或自己在并行化一个批处理循环，再让每次内层调用也扇出，会过度订阅机器、通常反而退化——让内层调用跑 `Serial`，把并行留在外层。（对一批乘积，优先用内置的[批量 GEMM](批量GEMM.md) 入口，它把整批作为一个整体来调度。）其三，**对确定性敏感的调试**，此时你想用单线程路径排除掉一个调度变量。

## 带宽受限的形状有自己的策略

矩阵-向量乘（`m == 1` 或 `n == 1`）以及其他内存受限的形状不是计算受限的，因此按工作量决定的工作线程数对它们是错的模型。这些路径使用一条独立规则：在一个由 LLC 导出的字节下限以下保持串行——单核已经拿到了完整的末级缓存带宽，拆分只会增加争用——而在其之上直接跨到一个带宽上限（逻辑核心数的一个分数，因为 DRAM 远在最后一个核之前就饱和了）。*少数几个*工作线程是带宽扩展曲线上最糟的点，因此该策略越过它、而非从中爬过。这是自动的；字节下限与上限可调（`GEMMKIT_GEMV_PARALLEL_BYTES`、`GEMMKIT_GEMV_THREAD_CAP`），完整论述见[小形状与 GEMV](小形状与GEMV.md)。
