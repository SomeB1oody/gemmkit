# 调优旋钮

gemmkit 里的每一个启发式——何时并行、何时打包某个操作数、一个形状从哪里起不再算“小”——都是一个带随库默认值的具名阈值，而不是写死的常量。这些默认值是在 Ryzen 9950X（Zen5）上标定的，其中少数几个在 aarch64 测出不同结果时按架构做了拆分，它们在大多数硬件上表现良好。若不然，每一个都能通过三种途径够到，无需改动源码。

## 解析顺序

一个旋钮在被读取的那一刻解析，取以下第一个已设置的层次：

1. **每次调用的参数。** 凡是有调用点等价物的旋钮，它直接胜出。最清楚的例子是并行度：你传给 `gemm` 的 [`Parallelism`](并行实践.md) 参数会覆盖任何全局线程策略。这一层位于 API 中，不在 `tuning` 里。
2. **编程式 setter。** `gemmkit::tuning::set_*(v)` 会无条件地存下一个值。一旦设置，之后的读取就再也不查环境变量。这是给那种在代码里自我调优的应用准备的：它理应压过部署环境所提供的任何东西。
3. **环境变量。** `GEMMKIT_*`。这是部署层：`source` 一份配置（例如 [gemmkit-tune](../gemmkit-tune/使用gemmkit-tune调优.md) 生成的那种），就能为某台主机重调一个已经构建好的二进制，无需重新编译。
4. **编译期默认值。** 那个标定好的常量，需要时按架构拆分。

setter 压过环境变量是有意的：调用了 setter 的应用就是选择了不理会环境；想让部署配置生效的应用，只要不去调那些 setter 即可。

环境变量**只在对该旋钮的首次访问时读取一次，随后缓存**为一个原子量。在某个旋钮首次被读之后再设置的值会被忽略，所以要在进程启动之前就导出配置。一个已设置但无法解析为非负整数的 `GEMMKIT_*` 变量会被当作拼写错误，而非无声的空操作：gemmkit 会在 stderr 上告警并回退到默认值（由于回退结果随后被缓存，告警每个旋钮只触发一次）。它绝不 panic，因为一个性能旋钮的拼写错误不应让进程崩溃。

## 旋钮清单

下面是覆盖所有 feature 与目标配置的完整清单；内部的 `tuning::knob_env_names` 注册表是它们的唯一真实来源。其中两个受 feature 或目标门控，只有编译进来时才存在。每个 getter 都有对应的 `set_*`；环境变量名就是 getter 名的大写形式加上 `GEMMKIT_` 前缀。

### 串行 / 并行门槛

| 环境变量 | Setter | 默认值 | 控制什么 |
| --- | --- | --- | --- |
| `GEMMKIT_PARALLEL_THRESHOLD` | `set_parallel_threshold` | 48*48*256 | `m*n*k` 低于此值时强制单线程。这是串行到并行的盈亏平衡点；线程池 fork 代价高就调高它，线程廉价且有值得拆分的小乘积就调低它。 |

### 打包门槛与步长

| 环境变量 | Setter | 默认值 | 控制什么 |
| --- | --- | --- | --- |
| `GEMMKIT_RHS_PACK_THRESHOLD` | `set_rhs_pack_threshold` | 2048 | 仅当 `m`（有多少行块复用它）超过此值时才打包 RHS 宏面板；低于它时就地读取 B。 |
| `GEMMKIT_LHS_PACK_THRESHOLD` | `set_lhs_pack_threshold` | 1024（aarch64：256） | 仅当每工作线程的列复用超过此值时才打包 LHS。打包在 aarch64 上更便宜，所以那里从更低的复用起就划算。 |
| `GEMMKIT_LHS_PACK_STRIDE` | `set_lhs_pack_stride` | 0（自动） | 字节门槛：列主序 A 若每步深度步长 `csa * sizeof(Lhs)` 达到此值，就打包以躲开对 TLB 和缓存不友好的跨步读取，与复用无关。`0` 表示从操作系统页大小推导。与下面的跨度和复用门槛取“与”：步长、跨度、复用三者须同时成立，强制打包才会触发。 |
| `GEMMKIT_LHS_PACK_SPAN` | `set_lhs_pack_span` | 0（自动） | 步长门槛的地址跨度伴侣：只有当整条深度切片遍历（`csa * sizeof(Lhs) * kc`）也达到这么多字节时，页级步长才会强制打包列主序 A。跨度仍驻留在缓存内的页级步长只是重走温热的缓存行，就地读取比它本要付的打包更快。`0` 表示自动（4 MiB）。 |
| `GEMMKIT_LHS_PACK_REUSE` | `set_lhs_pack_reuse` | 128（aarch64：4） | 复用门槛，为强制打包的收益而非成本定价：只有当至少这么多条 `nr` 宽列块复用每个打包面板时（`min(n, nc) / nr`，向上取整），上面的步长与跨度门槛才会真正触发打包。瘦高形状（`m` 远大于 `n`）跨度巨大却列块很少，把一次昂贵的打包摊到过少的复用上并不划算；`0` 会去掉这道门槛，只让步长+跨度这对来决定。aarch64 上这笔账几乎反转——打包便宜、而就地遍历要跨 16 KiB 的页——因此 M4 Max 从 4 条列块复用起就想打包（`m = k = 4096`：`n = 32` +88%，`n = 256` +220%），只有 `n_nt = 2` 仍偏向就地。 |
| `GEMMKIT_SHARED_LHS_MNK` | `set_shared_lhs_mnk` | 8e9（aarch64：5e7；32 位：禁用） | 并行打包路径上共享 A 预处理的 `m*n*k` 门槛，它以一次 fork-join 屏障为代价消除各工作线程的冗余打包。与该门槛独立地，预处理从 16 个工作线程起也会开启——那个宽度上冗余打包的代价必然压过屏障。 |
| `GEMMKIT_PACK_TRANSPOSE_TILE` | `set_pack_transpose_tile` | 16 | 打包操作数为跨步布局时所用的缓存分块转置的条带长度，把逐元素 gather 变成分块拷贝。实数与复数打包器共用它。 |

### 特殊路径阈值

| 环境变量 | Setter | 默认值 | 控制什么 |
| --- | --- | --- | --- |
| `GEMMKIT_GEMV_THRESHOLD` | `set_gemv_threshold` | 无上限 | 当另一维为 1 时，为专用 gemv 路径给 `min(m, n)` 设上限。触发 gemv 的是形状而非规模，此值只是给它封顶。 |
| `GEMMKIT_SMALL_K_THRESHOLD` | `set_small_k_threshold` | 16（aarch64：8） | `k` 在此值及以下时，形状走通用 small-`k` 路线（单个深度面板、不打包），而非寄存器分块驱动器。 |
| `GEMMKIT_SMALL_MN_DIM` | `set_small_mn_dim` | 16 | `m` 与 `n` 都在此值及以下（且 `k` 较长）时走横向内积路线，每个输出是一次 SIMD 归约点积。`0` 关闭该路线。 |
| `GEMMKIT_SMALL_MN_PACK_MIN_K` | `set_small_mn_pack_min_k` | 16 | small-`m,n` 打包层的 `k` 门槛：跨步的小形状仅在 `k` 超过此值时才把失配的操作数拷入 `k` 连续的暂存。 |
| `GEMMKIT_GEMV_PARALLEL_BYTES` | `set_gemv_parallel_bytes` | 0（自动） | 带宽受限的 gemv/gevv 保持单线程的字节下限（低于它时被触及的数据驻留在 LLC，拆分只会亏）。`0` 表示从 LLC 大小推导：有 L3 时取单核可达 L3 的一半；无 L3 的 aarch64 机器取簇共享 L2 的八分之一（M4 Max 上即 2 MiB——档位池砍掉 fork 成本之后，并行从 3.2 MiB 的矩阵起就已获胜）。 |
| `GEMMKIT_GEMV_THREAD_CAP` | `set_gemv_thread_cap` | 0（自动） | 带宽受限的 gemv/gevv 可用的最大工作线程数，因为 DRAM 在远低于逻辑核数处就饱和。`0` 表示从核数推导一个代理值；在高带宽部件上可调高。 |
| `GEMMKIT_K_STREAM_MAX` | `set_k_stream_max` | 32 | axpy 形状的 gemv 在整个深度扫描中把输出面板保持在寄存器里的 `k` 上限；超过它则普通的列外层形式胜出。 |
| `GEMMKIT_SEQ_INTERNAL_BYTES_PER_WORKER` | `set_seq_internal_bytes_per_worker` | 128 KiB | aarch64 批量 GEMM 的分水岭：当每批工作线程的字节份额超过此值时，一个批元素跨机器拆分，而非一线程一个缓存热运行。仅在 aarch64 上被查询。 |
| `GEMMKIT_I8_VNNI_MIN_PAR_MNK` | `set_i8_vnni_min_par_mnk` | 768^3 | `m*n*k` 低于此值时，自动选中的 VNNI `i8` 内核会把*多线程*问题交给加宽兜底（VNNI 强制的 RHS 打包屏障在小并行问题上不划算）。与 VNNI 逐位一致。需要 `int8` feature。 |

### 调度粒度

| 环境变量 | Setter | 默认值 | 控制什么 |
| --- | --- | --- | --- |
| `GEMMKIT_PARALLEL_OVERSAMPLE` | `set_parallel_oversample` | 8 | 并行驱动器目标是每工作线程这么多个工作块，按需从共享游标领取。越高，负载均衡越细、尾部越小，但原子领取越多；越低，越粗糙但开销越少。 |
| `GEMMKIT_PAR_MNK_PER_WORKER` | `set_par_mnk_per_worker` | 2000000（线程化 wasm：262144） | 自动线程数的粒度：自动路径以 `m*n*k` 除以每工作线程这么多工作量为目标（再以核数与任务数封顶、下限为 1），使数量随总 flops 而非线性尺寸扩展。wasm 工作线程的启用成本远低于原生线程，因此 wasm 的下限更低。`0` 等同于 `1`（始终全宽）。 |
| `GEMMKIT_PACKED_OVERSAMPLE` | `set_packed_oversample` | 2 | 打包 LHS 路径的拆分目标（区别于上面的通用粒度）：拆得太狠会过频地重打包 A 从而回退，所以这个最优值更低。 |
| `GEMMKIT_POOL_CLASSES` | `set_pool_classes` | 2（aarch64：1；其余架构：0） | 满机器宽度之下的减半档位数——先半宽，再四分之一宽——gemmkit 为每个档位都保留一个私有的、持久化的 rayon 池，首次用到时惰性构建（每线程约 12 微秒），此后不再重建。自动路径把它的 worker 数精确卡在某个档位上，让 fork/join 屏障处没有线程闲置；显式的 `Rayon(n)` 仍然恰好拿到 `n` 个 worker，只是会跑在能容纳它的最小档位池里。`0` 关闭档位池，让每次调用都留在环境池上。封顶为 3。x86_64 默认 2 档，aarch64 默认 1 档（M4 Max 上 7 宽的半宽档位在 `128^3`-`224^3` 以 2.1-2.6 倍胜出，再加一个更小的档位反而会把这些形状错误路由过去）；其余架构默认关闭，等待设备实测。 |
| `GEMMKIT_FULL_WIDTH_MNK` | `set_full_width_mnk` | 0（自动） | `m*n*k` 高于此值时，自动路径才会离开其最大档位池、转向全机宽度——到那时，额外的全宽 worker 才终于能赚回它们所付的 fork/join 代价；低于它时，即便存在更多核心，自动路径也停留在其最大档位。`0` 表示按架构推导：x86 为 110_000_000（标定自 9950X），aarch64 为 14_000_000（标定自 M4 Max——全宽借助 E 核从 `256^3` 起反超）。`MAX` 让自动路径无条件钉死在最大档位，全宽永不介入。 |

### 分块上限

| 环境变量 | Setter | 默认值 | 控制什么 |
| --- | --- | --- | --- |
| `GEMMKIT_MC_REG_PANELS` | `set_mc_reg_panels` | 8 | A 宏面板被限制为这么多微 tile 行（`this * MR`），遵循 BLIS “MC 保持为 MR 的一个小倍数”的规则。 |
| `GEMMKIT_NC_NO_L3_PANELS` | `set_nc_no_l3_panels` | 512 | 无 L3 时（Apple Silicon 之类）的列块为 `min(this * NR, N)`。存在 L3 时此旋钮无效。 |
| `GEMMKIT_TINY_BLOCK_DIM` | `set_tiny_block_dim` | 64 | `m` 与 `n` 都在此值及以下的形状跳过完整的 BLIS 分块模型，只把 A/B 面板留在 L2。 |
| `GEMMKIT_KC` | `set_kc` | 512 | 小矩阵捷径里的深度块：`k` 钳到此值。 |
| `GEMMKIT_KC_MIN` | `set_kc_min` | 512 | 主模型的 `kc` 下限：L1 贴合的深度估计被抬升到至少此值，让小 L1 也不会让微内核的深度扫描挨饿。 |

### 深 K 与 wasm

| 环境变量 | Setter | 默认值 | 控制什么 |
| --- | --- | --- | --- |
| `GEMMKIT_DEEP_KC_BYTES` | `set_deep_kc_bytes` | 0（自动） | 深收缩路径的启用门槛，以字节计：窄输出家族（`f16`/`bf16`）在其单个 RHS 微面板（`nr * k * sizeof(N)`）超过此值时，从单面板形式切到 f32 输出的多切片孪生。`0` 表示从检测到的 L2 的一半推导。 |
| `GEMMKIT_PREFETCH_MIN_BYTES` | `set_prefetch_min_bytes` | 0（自动） | 驱动器 C tile 软件预取的启用门槛，以字节计：一旦一次调用的工作集（`A + B + C` 字节）超过此值，输出微 tile 便从 LLC 之外流入，驱动器就在每个微内核调用之前对相应微 tile 发出一次 T0 预取，掩盖其读-改-写延迟；低于此值时 tile 驻留缓存，这个提示纯属开销。`0` 表示从单核可达的 LLC 推导（有 L3 取 L3，否则取 L2）。非零值即字节门槛本身，所以 `usize::MAX` 关闭预取，`1` 强制开启。仅 x86_64（其他目标为空操作，因此 aarch64/wasm 不变）；对数值不可见，开或关逐位一致。 |
| `GEMMKIT_WASM_THREADS` | `set_wasm_threads` | 8 | 带线程 wasm 构建的工作线程数，因为 wasm 没有 `available_parallelism` 可查。它给 gemmkit 的 wasm rayon 线程池定尺寸。仅在带 `wasm_threads` feature 的 wasm32 上存在。 |

## 关于 GEMMKIT_FAST_TEST

你可能会在测试框架里看到 `GEMMKIT_FAST_TEST`。它把正确性扫描收窄以跑得更快，是一个**仅测试套件使用**的开关；库本身从不读取它，设置它对生产环境的 GEMM 没有任何影响。

## 超越手工调参

手动设旋钮，适用于你已经知道该动哪一个的场合。要为某台具体机器标定整套旋钮，就运行自动调优器：它在一组探测形状上扫描每个旋钮，并写出一份你在运行前 `source` 的 `GEMMKIT_*` 配置，无需重新编译。这是 [gemmkit-tune](../gemmkit-tune/使用gemmkit-tune调优.md) 一章的主题。
