# 运行时ISA分发

gemmkit 只发布一个引擎，运行的指令集是在程序启动时选定的，而不是在编译时。在笔记本上构建、拷到服务器上运行的同一个二进制，只要服务器支持 AVX-512 就会用上它；同一个二进制放到较老的机器上，则会悄悄退回到较窄的内核。你不需要挑选后端、不需要用 `cfg` 分门别类、也不需要为每台主机重新编译：第一次 GEMM 调用会探测 CPU 特性、缓存胜出的内核，之后每次调用都只是通过那个缓存指针做一次普通的间接调用。

## 后端阵容

一个构建带有哪些内核，取决于目标架构；实际运行哪一个，取决于 CPU。从快到慢，候选者依次是：

- x86-64 上的 **AVX-512F**，最宽的浮点内核。窄元素类型另有两个点积特化内核与之并列：面向 `i8 -> i32` 的 **AVX-512 VNNI**（`vpdpbusd`），以及面向 `bf16` 的 **AVX-512 BF16**（`vdpbf16ps`）。它们分别需要 `int8` 和 `half` feature，且 CPU 必须报告对应的特性位。
- x86-64 上的 **FMA / AVX2**，面向没有 AVX-512 的机器的加宽 FMA 内核。
- aarch64 上的 **NEON**，那里 SIMD 是基线（每颗 aarch64 CPU 都有），因此运行时无需探测。
- wasm32 上的 **simd128**，它在编译期而非运行期选定（见下文）。
- **scalar**，可移植的兜底路径。它在每个目标上都存在，当没有更好的选择时就由它接手。一个正确（尽管未加速）的结果始终可达。

Tile 几何是唯一随 `(元素类型, ISA)` 组合而变的东西：微内核计算一个 `MR x NR` 的寄存器 tile，其尺寸按 ISA 的向量宽度调校。对 `f32`，随库发布的 tile 是：

| ISA | f32 tile（MR x NR） |
| --- | --- |
| AVX-512F | 32 x 12 |
| FMA / AVX2 | 16 x 6 |
| NEON | 16 x 4 |
| simd128 | 8 x 4 |
| scalar | 4 x 4 |

这五个跑的是同一个通用浮点微内核，区别只在 tile 形状。`MR` 是 `MR_REG * LANES`，所以向量越宽，tile 就越高。`f64` 的 lane 数减半，因此 `MR` 也减半（AVX-512F 的 `f64` 是 `16 x 12`，以此类推）。VNNI 与 BF16 点积内核有各自按深度分组的几何，在[元素类型](元素类型.md)中介绍；这张表是背景知识，用于理解某个内核为什么这样打包和分块，而不是一个可供设置的旋钮。

## 自动选择

每种元素类型独占一个分发槽位，即一个持有类型化函数指针的 `OnceLock`。对该类型的第一次调用会让选择阶梯运行一次特性探测、挑出当前可用的最优内核，并存下它的单态化入口（普通、预打包、融合）以及 tile 几何。这一次性开销——`is_x86_feature_detected!` 探测与 `OnceLock` 初始化——由最先发生的那次调用承担；此后分发就是一次缓存指针加载加一次间接调用，在热路径上不再对 ISA 做任何分支。这背后没有 `transmute`，也没有原子指针的腾挪，只是每种类型一个类型化槽位。

有一点值得明说：**没有任何公开 API 会告诉你选中了哪个 ISA。** 这个选择是记忆化槽位的内部细节。如果你需要确认某个特定内核确实在跑，不要试图把它读回来，而应把它钉住（见下一节），让不匹配大声报错。

## 用 `GEMMKIT_REQUIRE_ISA` 钉住内核

设置环境变量 `GEMMKIT_REQUIRE_ISA` 会强制端到端只用某一个内核，而不再自动选择。可接受的取值（大小写不敏感，首尾空白会被去除）是：

| 取值 | 强制使用 | 同时接受 |
| --- | --- | --- |
| `scalar` | 可移植的标量内核 | |
| `fma` | FMA / AVX2 加宽内核 | `avx2` |
| `avx512f` | AVX-512F 加宽内核 | |
| `avx512vnni` | `i8` 的 `vpdpbusd` 点积内核（其他类型走普通 AVX-512F） | `vnni` |
| `avx512bf16` | `bf16` 的 `vdpbf16ps` 点积内核（其他类型走普通 AVX-512F） | `bf16` |
| `neon` | aarch64 的 NEON 内核 | |
| `simd128` | wasm32 的 `simd128` 内核 | `wasm` |
| `auto` | 正常的自动选择（未设置或为空时也是它） | |

`avx512vnni` 与 `avx512bf16` 只为它们各自的那一种窄类型选用点积内核，其余一切走普通 AVX-512F 路径，所以在这两个 pin 之下，混合负载中的其他类型仍能正确运行。

契约是**报错，而非回退。** 如果所请求的 ISA 不可用——因为 CPU 未报告该特性、因为取值命名了一个在当前目标架构上并不存在的 ISA（x86 上的 `neon`、aarch64 上的 `avx512f`），或者因为取值本身就是个拼写错误——分发会 panic，而不会悄悄跑另一个内核。这是有意为之，也正是 CI 想要的：一个目的就是要跑通 AVX-512 VNNI 路径的作业，绝不能因为特性标志拼错或模拟器配错，而在悄悄测试标量兜底的情况下通过。gemmkit 自己的 CI 就是这样钉住每个内核的：x86 的点积内核在 Intel SDE 下跑，NEON 在 aarch64 上跑，`simd128` 在 wasm 上跑；一个坏掉的 pin 会变成一次红色构建，而不是虚假的绿色。

这个值**只读取一次，在首次分发之前**，并与内核选择一起记忆化。请在任何 GEMM 运行之前把它设入进程环境；进程运行中途再改它不会有任何效果，因为槽位已经填好了。无法识别的取值会是硬错误，正是为了让它不会被误当成 `auto` 而蒙混过关。

## WebAssembly 是编译期决定的

wasm32 没有运行时特性探测，所以 `simd128` 不是靠探测机器选出来的；它由编译期 `cfg` 选定，而构建必须真的用 `-C target-feature=+simd128` 把它打开。忘了这个标志，wasm 构建就会悄悄用标量兜底。钉住 `GEMMKIT_REQUIRE_ISA=simd128` 会把这种无声降级变成断言：如果 SIMD 路径没在跑，构建就会 panic——这正是 wasm CI 作业钉住它的原因。完整的 wasm 构建流程，包括带线程的目标，见 [no_std 与 WebAssembly](no_std与WebAssembly.md)。
