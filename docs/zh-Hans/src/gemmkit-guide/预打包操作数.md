# 预打包操作数

在微内核触碰 A 和 B 之前，引擎会先把它们各自拷贝进一种对缓存友好的微面板（micropanel）布局：一块块连续的 tile，微内核以单位步长扫过。对于一次性的乘积，这次拷贝纯粹是准备工作，付一次就丢掉。但很多工作负载会一遍遍地乘同一个矩阵：一个线性层用同一个固定权重矩阵去乘一串激活批次，一个求解器用同一个算子去乘许多右端项。每次调用都重打包那个固定操作数，等于把已经做过的活白扔。预打包操作数 API 让你把打包只付一次，然后在所有共享该操作数的乘积之间复用。

## 打包右操作数

最常见的情形是固定的 `B`（权重）配上一串大小各异的 `A`（激活）。调用一次 [`prepack_rhs`](https://docs.rs/gemmkit)，把 `k x n` 的 `B` 变成一个 [`PackedRhs`](https://docs.rs/gemmkit) 句柄，之后每次乘积都把这个句柄交给 `gemm_packed_b`：

```rust
use gemmkit::{prepack_rhs, gemm_packed_b, MatRef, MatMut, Parallelism};

// 固定权重：一个 k x n 矩阵，被许多激活批次复用
let (k, n) = (512, 256);
let weights = vec![0.0f32; k * n];
let packed = prepack_rhs(MatRef::from_col_major(&weights, k, n));

// 每个激活批次：一个 m x k 输入，共享已打包的权重
let mut c = vec![0.0f32; m * n];
gemm_packed_b(
    1.0,
    MatRef::from_row_major(&input, m, k),
    &packed,
    0.0,
    MatMut::from_col_major(&mut c, m, n),
    Parallelism::Rayon(0),
);
```

`prepack_rhs` 接受任意布局的 `B`，并透过步长来读取它，所以行主序、列主序或转置视图打包出来的结果都一样。打包在 `prepack_rhs` 内部单线程地跑一次；之后每次 `gemm_packed_b` 都跳过它。这块缓冲区记录了它构建时所用的分块几何（`nr`、`kc`、`nc`），消费方的调用会原样读回这套几何，因此一个面板永远是按它自己的 tiling 来解读的。由于该缓冲区在 GEMM 期间只读、从不写，一个 `PackedRhs` 可以安全地跨线程、跨并发调用共享，无需任何同步。`PackedRhs::rows()` 返回原始的 `k`，`cols()` 返回原始的 `n`。

这个句柄对任何 `(k, n)` 与所打包 `B` 相符、且输出 `C` 为**列主序倾向**（`|csc| >= |rsc|`）的乘积都有效。最后这条约束是唯一的意外。一个行主序的 `C` 会让引擎在内部交换 `A` 与 `B` 以保持存储连续，而预打包的 `B` 无法挪到 `A` 的角色上，所以 `gemm_packed_b` 遇到行主序的 `C` 会 panic，并让你对那种布局回到普通的 `gemm`。`A` 的布局不受约束；被钉住的只有 `C`。

在固定配置下，`gemm_packed_b` **复现**普通的 `gemm`，且结果对 worker 数确定，只有一个很窄的例外：对极小的乘积（`m` 与 `n` 都在 `tiny_block_dim`、默认 64 及以下）以及 gemv 形状（`m == 1` 或 `n == 1`）的乘积，两者可能在最后一个 ULP 上不同。原因在于路由，而非误差：普通 `gemm` 会把这些形状改道到某条[特殊路径](小形状与GEMV.md)，而预打包入口始终驱动通用的打包内核。两个答案都正确；只是恰好在那些本会被特殊路径接管的形状上，它们的求和次序略有不同。

## 左操作数的镜像

对称的情形——固定的 `A` 配上一串变化的 `B`——由 [`prepack_lhs`](https://docs.rs/gemmkit) 产生一个 [`PackedLhs`](https://docs.rs/gemmkit)、再由 `gemm_packed_a` 消费来服务。它与 RHS 那一对完全镜像，只是轴换了标签：`PackedLhs::rows()` 是原始的 `m`，`cols()` 是共享的 `k`。

在内部，LHS 打包并不是一条独立的代码路径。由引擎的 A/B 对称性，一个预打包的 `A` 恰好就是转置乘积 `C^T = B^T A^T` 的预打包 `B`，所以 `prepack_lhs` 铺下的是完全相同的微面板缓冲区，只是把维度记成 LHS 的说法。可见的后果是 `C` 布局约束翻转了：`gemm_packed_a` 要求 **行主序倾向** 的 `C`（`|csc| <= |rsc|`），与 RHS 入口正好相反，因为列主序的 `C` 会让 `A` 保持在真正的 LHS 角色上，而一块转置 RHS 缓冲区填不了这个角色。`C` 是行主序时选打包 `A` 的入口，是列主序时选打包 `B` 的入口；两者合起来覆盖了两种取向。

## 融合变体

每个打包入口都有一个融合孪生，位于 `epilogue` feature 之后：`gemm_packed_b_fused` 与 `gemm_packed_a_fused` 会在打包内核本就要执行的那次存储里，加上逐行或逐列的偏置和一个可选的激活（偏置与激活类型见[融合 Epilogue](融合Epilogue.md)）。**同一个** `PackedRhs` 或 `PackedLhs` 句柄同时服务普通入口和融合入口：epilogue 只作用于存储侧，从不触碰打包，所以你只构建一次缓冲区，每次调用再决定是否融合。有两点是打包路径特有的。其一，与普通 `gemm_fused` 不同，打包融合入口从不被改道到 gemv / 小 `m,n` / 小 `k` 内核；它们始终驱动通用打包内核（与普通打包入口所记录的分歧相同）。其二，逐行 / 逐列偏置始终以自然的**用户**坐标系给出：`gemm_packed_a_fused` 替你处理内部转置，所以无论调用哪个入口，`PerRow` 偏置的长度都是 `A.rows`。

## 预打包 i8 权重

在 `int8` feature 之下，这套模式延伸到量化推理：`prepack_rhs_i8` 把固定的 `i8` 权重矩阵打包成 `PackedRhs<i8>`，由 `gemm_i8_packed_b` 消费（`i8` 输入、`i32` 输出）。预打包在这里比对浮点收益更大，原因是结构性的。AVX-512 VNNI 内核（`vpdpbusd`）从一种 k-四元组交错布局里读取它的 RHS，而这种布局无法就地生成，因此它的 RHS 打包在**每次调用时都是强制的**；在小 `m` 下，这个逐调用的 `O(k*n)` 打包很容易压过 `O(m*k*n)` 的计算量。预打包把它彻底从热循环里删掉。这块打包缓冲区还**钉住了内核选择**：它是为进程分发所选定的那个整数内核（VNNI 交错，或加宽内核的普通面板）而铺设的，而 `gemm_i8_packed_b` 始终跑那同一个家族，所以缓冲区绝不会被错读。整数累加是精确且与 ISA 无关的，因此对每个合法形状，打包路径与普通路径**逐位一致**，完全没有小形状的例外。

```rust
use gemmkit::{prepack_rhs_i8, gemm_i8_packed_b, MatRef, MatMut, Parallelism};

let packed = prepack_rhs_i8(MatRef::from_col_major(&weights_i8, k, n));
let mut c = vec![0i32; m * n];
gemm_i8_packed_b(
    1,
    MatRef::from_row_major(&input_i8, m, k),
    &packed,
    0,
    MatMut::from_col_major(&mut c, m, n),
    Parallelism::Rayon(0),
);
```

## 什么时候值得预打包

预打包是用一次前置的 `O(k*n)` 拷贝，换取此后针对该操作数的每次乘积都省下重打包，所以它恰好在操作数被复用到足以摊销这次拷贝时才划算。一次性的乘积，或每次调用都变化的操作数，一无所获：此时打包纯属额外开销，普通 `gemm` 才是正解。还要留意，浮点路径本就不一定会打包 `B`：在小 `m` 下普通 `gemm` 会就地读取未打包的 `B`（由 `rhs_pack_threshold` [旋钮](调优旋钮.md)控制），所以给一个复用不多的浮点 `B` 做预打包甚至可能亏。最清晰的收益，是这套 API 得名于的固定权重推理循环，尤其是 `i8` VNNI 路径——它的 RHS 打包本来每一次调用都躲不掉。拿不准时就把循环两种写法都测一测；交叉点取决于你的复用次数和你的机器，而非某条固定规则。

裸指针形式（`prepack_rhs_unchecked`、`gemm_packed_b_unchecked`，以及它们的 `_with`、LHS 与 `i8` 对应版本）是为自行校验输入的适配器和 FFI 而设的；见 [Unchecked 层](Unchecked层.md)。
